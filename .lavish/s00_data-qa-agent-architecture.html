01 · Recommended stack at a glance

🖥️ Frontend

React + Vite (TypeScript)

Chat-style SPA. MSAL for login. Served as a container or Static Web App.

⚙️ Backend API

FastAPI (Python, uv)

Validates JWTs, enforces RLS context, orchestrates the agent. Matches your CLAUDE.md toolchain.

🤖 Data Agent

Pydantic AI · pluggable LLM

Native tool-calling (ReAct-style) loop + per-user memory. Structured output, traced with Logfire. Model provider is swappable (Claude, DeepSeek, …).

🗄️ Database

Azure PostgreSQL Flexible Server

PG 16+. Row-Level Security enforces who sees which rows. The single source of truth.

🔐 Identity

Microsoft Entra External ID

OIDC login, MFA, social/email sign-in. Issues JWTs the backend trusts.

☁️ Compute

Azure Container Apps

Serverless containers, scale-to-zero, built-in ingress + revisions. No cluster to babysit.

🔀 Data pipeline

dbt + ingestion (CSV→Postgres)

dbt models/tests/docs turn raw loads into clean tables; the agent reads dbt's schema to write SQL.

Why this shape: Container Apps is the 2026 default for serverless microservices without running Kubernetes; FastAPI + React + PostgreSQL 16 is the mainstream modern data-app stack; Pydantic AI + Logfire is already wired into your skills. A bare-bones 3-service setup with Postgres + registry lands in the ~$25–60/month dev-test range (scale-to-zero keeps idle cost low).

05 · Auth + row-level security — the heart of it

Three layers stack so data isolation holds even if app code has a bug: authentication (who are you), authorization (what role), and row-level security (which rows) enforced by the database itself.

Layer 1 · AuthN

Entra issues a JWT

User logs in via Entra External ID. Frontend gets a signed access token carrying sub (user id) and roles. Backend verifies it against Entra's public keys on every request.

Layer 2 · AuthZ

Role check in the API

FastAPI dependency reads roles/claims and gates endpoints (e.g. analyst vs admin). Coarse "can you call this at all" checks live here.

Layer 3 · RLS

Postgres filters rows

Before any query, the API sets a session variable with the user id. Policies on each table compare it to the row's owner — the DB returns only permitted rows, automatically.

 Enable RLS and scope every read to the current session user
ALTER TABLE insights ENABLE ROW LEVEL SECURITY;
CREATE POLICY tenant_isolation ON insights
  USING (owner_id = current_setting('app.current_user_id')::uuid);
 API sets the context per request (inside a transaction):
SET LOCAL app.current_user_id = '8f3a...e21';
One thing to confirm: your brief said "row level control so logged-in users can see other users' rows." Standard RLS does the opposite — it isolates each user. I've assumed you meant isolation by default, with roles (e.g. admin/analyst) that can see across users. Confirm direction in Decision A.

06 · The data agent — "automating data science"

This is the part that makes the demo worth building. A Pydantic AI agent receives the question + the DB schema, plans an approach, calls tools to query and analyse, and returns a typed answer with an optional chart spec. Logfire traces every step so you can see it think.

Agent tools (v1)

  • 🔎 get_schema() — curated tables/columns from dbt docs (manifest/catalog)
  • 🧮 run_sql(query) — execute read-only SQL under RLS, row-capped
  • 📊 make_chart(spec) — return a Vega-Lite/Recharts spec for the UI
  • 📈 profile_column(table, col) — quick stats / distributions
  • 🧠 recall() / remember(fact) — read & write the user's long-term preferences

Guardrails (non-negotiable)

  • 🔒 Agent connects as a read-only DB role — no writes, ever
  • 🧱 RLS still applies — agent can't escape the user's data scope
  • ⛔ SQL allowlist: SELECT only, parsed + validated, statement timeout
  • 💸 Token + row caps to bound cost and blast radius
DB access tools: for v1 use in-process typed tools (asyncpg/SQLAlchemy) — cleanest control of the read-only role, per-request RLS session variable, SELECT allowlist, and timeouts. An MCP Postgres server is a good later option if you want to reuse the DB tool across agents/IDEs, but you'd still wrap it in the same guardrails. See the FAQ (§15).

Your questions on the agent — answered

ReAct & tools

No need for LangChain to get ReAct. Pydantic AI runs a native tool-calling loop — the modern form of ReAct — with typed tools, structured output, dependency injection, and streaming. It's model-agnostic and already wired into your skills + Logfire.

Swap to DeepSeek

A config change, not a re-architecture. The model lives behind the agent, so DeepSeek plugs in via its OpenAI-compatible endpoint. Keep the provider abstracted and A/B for SQL accuracy vs cost. Note: confirm the exact model id — DeepSeek's documented ids are deepseek-chat / deepseek-reasoner; I couldn't verify deepseek-v4-flash.

Learns preferences

Memory is a data concern, not a framework one. Add an agent_memory table in Postgres + pgvector (supported on Azure PG Flexible). The agent recall()s relevant prefs at the start and remember()s new ones at the end — all under RLS, so a user's memory is isolated like their data.

Recommendation: stay on Pydantic AI, keep the model provider pluggable, and add memory via Postgres + pgvector — you get ReAct tool-use, provider choice (Claude/DeepSeek), and a learning agent without taking on LangChain. Prefer LangGraph only if you specifically want its graph/checkpointer persistence model — see Decisions F–I.

07 · Data pipeline & the Q&A engine

A good answer starts with good data. CSVs land in Postgres, dbt turns them into clean, tested, documented tables, and the agent reads dbt's own schema docs to write accurate SQL. That dbt-docs → agent link is the actual Q&A engine.

dbt vs Pydantic — not rivals; they own different layers. dbt owns the data model (raw → clean, tested, documented tables). Pydantic AI owns the agent (reads dbt's docs, writes SQL, answers). Pydantic the library owns validation of API/agent I/O. The "single solution" for pipelines is dbt — it doesn't overlap the agent.

1 · Ingest (EL)

CSV → raw tables

dbt doesn't load files — pair it with a small Python dlt pipeline (or a COPY loader) that lands each CSV into a raw schema. dbt seed is fine for tiny static reference files.

2 · Transform (dbt)

raw → staging → marts

dbt-core (Postgres adapter) models the data in SQL, adds tests (not-null, unique, relationships) and descriptions. You get lineage plus versioned, reviewable transforms.

3 · Serve to agent

dbt docs → SQL

The agent's get_schema() reads dbt's manifest.json/catalog.json — table & column descriptions become the model's context, so generated SQL is grounded. Queries still run under RLS.

Runs as a scheduled/triggered Container Apps Job (ingest → dbt build). For v1, trigger on new CSV or a cron; wire it into CI later.

users:
  - email: admin@example.com
    role: admin      # sees all rows
  - email: user1@example.com
    role: user       # housing access
  - email: user2@example.com
    role: user       # no housing access

How it flows

  • Pipeline reads datasets.yaml → dlt ingests each csv → raw.
  • dbt models it into marts; a row is upserted into the datasets registry.
  • access populates dataset_access → RLS enforces who can query it.
  • Agent lists datasets the user may see, then answers over them.
App config (DB URL, model keys, provider choice) via pydantic-settings — reads .env locally and Key Vault in Azure, as one typed Settings object.

🌱 Environments — dev now, staging/prod later

v1 runs entirely as dev. Everything is env-parameterized from day one, so adding staging/prod later is new config + IaC values, not a rewrite.

  • APP_ENV selects config via pydantic-settings — .env now; .env.staging / .env.prod later.
  • Azure resources carry an env suffix: dataqa-dev-* today → dataqa-staging-*, dataqa-prod-*. One IaC module, invoked per env.
  • Databases are split by environment, never shared — one Postgres server for dev now; separate staging/prod servers later. Schemas stay identical within each.
  • CI deploys dev on merge now; promotion to staging/prod (branch/tag-gated) is added later.
Naming heads-up: the dbt staging schema (a data-modeling layer) is unrelated to a staging deployment environment. Same word, different thing.

💻 Yes — the whole app runs locally

One command boots everything on localhost; nothing in Azure is required to develop and demo.

docker compose up        # postgres+pgvector, backend-api, data-agent, frontend
make seed                # 3 users + register datasets
make pipeline            # dlt ingest data/incoming/*.csv, then dbt build
# open http://localhost:5173
  • Auth: Entra is cloud-only, so local uses a dev-auth stub that mints a signed dev JWT as admin/user1/user2 (AUTH_MODE=dev). Flip config to use real Entra — no code change.
  • Config/secrets: .env instead of Key Vault. Data, RLS, memory, dashboard, evals: all run locally.
  • Only outbound call: the LLM API (needs an Anthropic/DeepSeek key). Want fully offline? Point the provider at a local model (e.g. Ollama) — a config swap, thanks to the abstraction.

11 · Evaluation & user-journey tests

A living suite of journeys — login → ask → agent answers — that you extend by adding YAML. Runs locally and in CI so the agent can't silently regress.

📋 evals/journeys.yaml (add freely)

- name: user1 asks housing avg price
  as_user: user1
  question: Average sale price by suburb?
  expect:
    - agent_answers_successfully
    - sql_is_select_only
    - mentions: [suburb, price]

- name: user2 is isolated from housing
  as_user: user2
  question: Show housing sales
  expect:
    - no_rows_or_access_denied   # RLS

How it runs

  • A pytest harness loads each journey, authenticates as the named user, calls the real /ask flow, and checks the expectations.
  • Uses pydantic_evals to score answers (success, SQL-only, expected mentions) with structured evaluators.
  • Doubles as your RLS isolation test — user2 must never see user1's rows.
  • Runs in Phase 5 CI; failures block deploy. Add a case = add coverage.

15 · Platform & portability FAQ

Your questions about cloud lock-in, LLM services, containers, and the database — answered.

Can I easily swap Azure for AWS or Google?

Mostly yes — the app code is portable (containers, FastAPI, React, Postgres, dbt, Pydantic AI). Only the managed services change. Use Terraform instead of Bicep if portability matters, and keep 12-factor config so you swap infra, not code.

PostgresFlexible ServerRDS / AuroraCloud SQL
IdentityEntra External IDCognitoIdentity Platform
SecretsKey VaultSecrets ManagerSecret Manager

Container Apps ↔ Cloud Run ↔ App Runner are close to 1:1 for containers, so that swap is small.

Does Azure have an LLM service (AI Foundry)? Is it cross-cloud?

Yes — Azure AI Foundry (+ Azure OpenAI) is Azure's model service. But binding to it couples you to Azure. Cross-cloud portability comes from your model abstraction (Decision G), not the cloud's service.

The same Claude model is reachable via the Anthropic API (cloud-neutral), AWS Bedrock, or Google Vertex — Pydantic AI just points at a different gateway. Recommendation: use the direct Anthropic API (or a neutral gateway) so the LLM isn't tied to where your compute runs.

Is a compute container 1-for-1 with a Docker image?

Yes. One image per service. A Container App runs that one image across N replicas (scale-to-zero to many). v1 = 3 service images (frontend, backend-api, data-agent) + 1 job image (data-pipeline) = 4 images in the registry.

Can there be just one Azure Postgres DB, or multiple?

One Flexible Server, one database, multiple schemas: raw / staging / marts (dbt) plus app (users, datasets, conversations, events, …). agent_memory is a pgvector table in that same DB — not a separate server. Split into more instances later only if scaling demands it.

Best way to implement DB-access tools — does MCP for Postgres make sense?

For v1, in-process typed tools (asyncpg/SQLAlchemy) are best — cleanest control over the read-only role, per-request RLS session variable, SELECT allowlist, and statement timeouts.

MCP Postgres makes sense later if you want to reuse the same DB tool across multiple agents/IDEs or run it out-of-process — but you'd still enforce the same guardrails around it. Not needed for v1, and it adds a network hop.

Later: can I add an Android app, or put the agent in a Slack channel?

Yes — and cheaply, because the backend-api + data-agent are headless (a clean /ask API behind a JWT). The web React app is just one client. New surfaces are new clients, not rewrites — they reuse the same agent, RLS, memory, datasets, and pipeline.

  • 📱 Android app — native (Kotlin) or React Native/Flutter — calls the same /ask API and signs in with Entra's native OIDC + PKCE mobile flow.
  • 💬 Slack bot — a thin Slack app (Bolt / Events API) catches a slash-command or mention, calls the same /ask, and posts the answer back in-channel.

The one design task per surface is identity mapping — link the surface's user (Slack user, mobile login) to a users row so RLS and per-user memory still apply. Everything else is shared.

This is the payoff of Decision C (3 services): an API-first core means multi-surface is a v2 add-on, not a re-architecture.
