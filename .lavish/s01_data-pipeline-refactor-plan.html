Pipeline Refactor Proposal dbt · staging vs marts Rev 4 — marts redefined

Query the staging grain, not the datamart

Today the agent can only see marts.* — three heavily pre-aggregated tables that collapse every sale/bond into a single first-year-vs-last-year comparison per postcode. This is v4: your clarification changes the target architecture, not just the column list. Staging stays record-grain and as wide as relevant — a clean mirror of raw. Marts stop pre-calculating growth/yield entirely and become clean summary building blocks (sum, count, median by postcode × property_type × month) — the agent computes growth, rolling averages, and yield itself from those, at any window it needs.

Direct answer Yes, raw is a latency (and correctness) problem — but this is no longer "staging vs. marts." It's staging for record-level lookups, and rebuilt marts as the agent's calculation building blocks.

staging.* stays record grain and gets widened to "as much as relevant" — it's the clean, verified mirror of raw, used when a question is genuinely about individual records (e.g. "top 10 addresses by sale price in a suburb"). marts.* gets rebuilt as small (~120k row) summary tables — n_sold, total_sale_value, median_price by postcode × property_type × month — and the agent writes the SQL for growth, rolling averages, and yield on top of them. The expensive part (aggregating ~3M rows) still only happens once, at build time; what the agent computes live is cheap arithmetic over an already-small table.

01 · Current state — what the agent can see today

Three schemas, one boundary. raw and staging are pipeline-internal today — the agent's DB role (agent_ro) has no GRANT on either schema, so it physically cannot query them even if the LLM tried. Everything the agent knows comes from get_schema() reading the dbt manifest filtered to schema == "marts" (services/data-agent/agent/schema.py:80).

What's already lost by the time marts land

  • Every year except the first and last (no real trend line, no rolling average)
  • Month/day resolution — both source date fields already carry it, staging discards it to year
  • Individual sale records — address, exact price, exact date — nothing below postcode survives
  • area_sqm and zoning — never selected from raw.sales into staging today, see §02
  • Bedroom count — carried in stg_rent.bedrooms but dropped before the marts

Questions the agent cannot answer today

  • "10-year growth" or "12-month rolling average" — marts only ever store one fixed first-vs-last window
  • "Top 10 addresses by sale price in a suburb" — no record-level table
  • "2-bedroom vs 3-bedroom rent" — bedrooms already dropped
  • "Average price by lot size" — area_sqm never reaches staging

Why this happened

Reasonable v1 default — the marts were designed around one query shape ("rank postcodes by growth") and the dbt tests (assert_*_has_coverage) were written to verify exactly that shape. Nobody over-engineered; the schema just hasn't grown past its first use case yet.

03 · The latency evidence — measured, not guessed

Numbers below are from this repo's actual full-size local CSVs and existing dbt comments, not estimates.

Expensive

Record grain: ~3.1–3.3M rows per dataset

Measured directly: wc -l data/nswgov_df.csv → 3,089,683 lines; wc -l data/rentboard_df.csv → 3,305,671 lines. This matches the pipeline's own comment: services/data-pipeline/dbt/profiles.yml:29 says staging is materialized as tables, not views, specifically because "views get inlined per reference, which re-runs the full percentile_cont aggregation over the raw ~2.7M rows every time." The same reasoning is why the redesigned marts (§05) still precompute sum/count/ median once at build time — the agent's own SQL only ever runs against the small result, never against 3M rows directly, for growth/yield/rolling-average questions.

Cheap

Precomputed grain stays cheap even at monthly resolution

services/data-pipeline/dbt/tests/assert_yield_pct_in_sane_range.sql:4 — a real dbt test comment — states the observed range was checked "across 10k+ postcode/type/year rows." Re-cut to postcode × property_type × month, that's an estimated ~120k+ rows (10k+ annual buckets × 12) for each redesigned summary mart — still two orders of magnitude smaller than record grain, and still fast.

Missing today

No statement_timeout is actually set

AGENTS.md's guardrail summary says "read-only DB role · RLS always applies · SELECT-only allowlist with parse validation and statement timeout · token + row caps." Reading services/data-agent/agent/db.py:34-41: it sets the RLS session var (set_config('app.current_user_id', ...)) but never sets statement_timeout. §06's retry budget makes this more urgent, not less: a bad query can now be attempted more than once.

Correctness, not just speed

raw.* is unsafe to hand an LLM directly

stg_sales.sql's own comments document hard-won fixes: dates arrive as YYYYMMDD.0 strings, postcodes need split_part(..., '.', 1), and a $10,000 price floor exists because non-arms-length family transfers otherwise produced "17 same-year 'sales' from $200 to $140,000" that skewed a bucket's median. If the agent queries raw.sales directly, the LLM has to silently re-derive all of that per question — a missed filter doesn't error, it just produces a confidently wrong number.

06 · Teaching the agent to validate and retry

You flagged this directly: "we lean on agent's SQL but it needs to be able to validate it and have multiple attempts to fix any syntax errors or re-get the data if it doesn't look as expected." This matters more now that the agent is writing real calculation SQL (§05), not just picking a pre-shaped query. Checked the actual code — today's behavior is close to the opposite of what you asked for.

What actually happens today (services/data-agent/agent/llm_agent.py:106-122)

The run_sql tool's attempt counter (sql_calls) only increments on success. So a syntax error already gets unlimited silent retries (bounded only by pydantic-ai's own step limit, not a deliberate design) — but the instant a query succeeds, even with zero rows or an obviously wrong shape, the very next call is hard-blocked: "STOP: do not call run_sql again." The system prompt reinforces this — it explicitly tells the model, on zero rows, to just report "the data isn't available" rather than investigate.

flowchart TD A["Agent writes SQL"] --> B["run_sql executes it"] B -->|"syntax / SQL error"| C{"Attempts remaining?"} B -->|"success, but zero rows or looks wrong"| C B -->|"success, looks right"| D(["Agent answers from this result"]) C -->|"yes"| A C -->|"no — budget exhausted"| E(["Agent answers with a caveat, or says it couldn't get reliable data"])

Proposed: a bounded attempt budget, not a one-shot gate

Replace sql_calls >= 1 with a configurable max_sql_attempts (e.g. 3), incremented on every call — failure or success — not just success. Each tool response carries how many attempts remain. The system prompt's "give up on zero rows" line gets rewritten to: check the filters/join keys and retry if attempts remain.

This directly trades against the plan's original topic: latency

A 3-attempt budget means a single user question can now cost up to 3 DB round trips and 3 LLM turns in the worst case, not 1. That's why the statement_timeout fix (§03, §07) matters more with this change than without it.

08 · Risks & open questions

This is a real breaking change, not an addition

§07's "Migrate the offline stub + tests" item touches the no-API-key fallback path (nl2sql.py) that's documented as the app's "always answers" guarantee, plus 6 dbt tests and the smoke test. Underestimating this is the biggest execution risk in the whole plan — size it properly before committing to a timeline.

Sum/count-based growth is less outlier-robust than the median it replaces

Today's percentile_cont median was a deliberate choice — robust to a handful of outlier sales. Growth/rolling-average computed from total_sale_value / n_sold is a mean, which one $8M sale in a thin postcode can swing noticeably. Median stays in the summary marts for "typical price" questions; only the composable metrics (growth, rolling average) use the mean — worth stating that distinction explicitly in the agent prompt so it doesn't blur the two.

Retry budget adds worst-case latency and cost

§06's fix directly trades against this plan's original question. Up to max_sql_attempts× the DB round trips and LLM turns per question in the worst case. Needs a real number from §07's benchmark before picking the budget size (§09 Decision 3).

Monthly buckets will often be thin

A small postcode might see 0–2 sales in a given month. Mitigate by always surfacing n_sold/n_rented next to any monthly figure and teaching the agent to say "not enough data that month" rather than report a shaky number.

Full street address exposure needs a licensing glance

house_no/street_name/unit_no (§02, reversed to "bring in") would let the agent print a full address. Public Valuer-General data, not PII, but worth a quick terms-of-use check before shipping.

A richer schema doesn't guarantee better SQL

Teaching the agent calculation patterns (§05) and giving it retries (§06) raises the ceiling but not the floor — it can still write a wrong formula, just get more chances to. The eval journeys in §07 matter as much as the guardrails do.
