Feature Proposal frontend · new "SQL Editor" tab Databricks-style, governed by our RLS Rev 1 — for review

A governed SQL editor, right beside the chat

Add a third tab to the app — a Databricks-style SQL editor where a user can write and run SQL directly against the marts, browse the schema, see results as a grid or chart, and reuse the history. The twist that makes it fit this product: it runs through the exact same read-only, RLS-scoped, audited path the agent already uses — not a new door into the database. It's the agent's guardrails with a keyboard.

Direct answer Yes — and most of the hard parts already exist. We surface the agent's governed SQL executor as a first-class editor UI, rather than building a new query engine.

The data-agent already exposes run_select(sql, user_id) (services/data-agent/agent/db.py:30) that runs a query as the strictly read-only agent_ro role, under Row-Level Security, with a SELECT-only allow-list, a statement timeout, and a row cap. The chat UI, the results table, Vega-Lite charts, and the query_runs audit trail all exist. This feature is mostly wiring: a new backend endpoint that reuses that executor, a schema-catalog endpoint, and a new React editor tab. The net-new risk surface is small and lands in one governed place.

01 · What exists today

The frontend (frontend/src/App.tsx) has two views toggled by one button: Chat and, for admins only, Admin. A question flows to backend-api POST /ask, which sets the RLS context and proxies to data-agent POST /agent/ask. The agent turns it into SQL and executes it via run_select. The SQL editor slots into this exact spine — it just skips the natural-language step.

Already built — we reuse

  • run_select() — read-only, RLS, SELECT-only, timeout, row cap
  • validate_select() guardrail (sql_guardrails.py)
  • Results table + VegaChart renderer
  • query_runs audit table + admin dashboard
  • get_schema() reads the dbt manifest (real table/column docs)

Net-new — we build

  • A code editor component (syntax highlight, run shortcut)
  • POST /sql (backend) → POST /agent/sql (agent)
  • GET /schema/catalog — structured schema for the browser
  • New "SQL Editor" tab + a proper tab bar
  • CSV export, query history, (later) AI assist

The governing rule

The editor must execute through agent_ro (read-only), never through backend-api's app_user role, which can write. Same RLS, same guardrails as the agent. So user2 (no data grants) typing a valid query still gets zero rows — the editor demonstrates isolation, it doesn't bypass it.

04 · How it wires in

One new request path that mirrors /ask but skips NL→SQL. The user's raw SQL goes frontend → backend-api POST /sql (JWT + RLS + audit) → data-agent POST /agent/sql → run_select (read-only agent_ro, guardrails, RLS). Same executor, same blast radius as the agent.

05 · Editor library — a real choice to make

CodeMirror 6

Recommended
  • ~200 KB gzipped; tree-shakeable, Vite-native
  • @codemirror/lang-sql gives SQL highlight + schema-aware completion out of the box
  • Easy to theme to the app's dark palette
  • Simple React integration (small wrapper, no iframe/worker plumbing)

Cost: fewer "IDE" bells (no built-in command palette).

Monaco

Alternative
  • It is VS Code's editor — most "Databricks-identical" feel
  • Heavy: ~2 MB+, web workers, more Vite config
  • Completion needs a custom provider wired to our catalog anyway
  • Overkill for a single-statement SELECT box

Pick this only if fidelity to Databricks' look is a hard requirement.

Recommendation: CodeMirror 6 — the bundle-size and autocomplete story fit a lean app; you get 90% of the feel for 10% of the weight. Confirm in §11.

06 · Phased build

Each phase ships something runnable, mirroring the repo's pause-after-each-phase habit. Phase A is the whole MVP loop; B is productivity; C is the on-brand AI layer that reuses the agent.

Phase A

MVP loop

Write SQL → run → see rows, governed & audited.

  • 3-tab nav (Chat · SQL Editor · Admin)
  • CodeMirror editor + Run (⌘⏎)
  • POST /sql + POST /agent/sql
  • Results grid, errors, truncation notice
  • Schema browser via GET /schema/catalog
  • Audit to query_runs + events
  • pytest: guardrail reject, RLS isolation, row cap

Phase B

Productivity
  • Schema-aware autocomplete
  • Query history (from query_runs, click to reload)
  • CSV export
  • Client-side result sort / filter
  • Multiple draft query tabs (localStorage)
  • Grid ⇄ Chart toggle (reuse VegaChart)

Phase C

AI (on-brand)
  • "Ask AI" — NL → SQL into the editor (reuse agent, return SQL without auto-run)
  • Explain / Fix / Optimize the selected SQL
  • NL result filter (optional)
  • "Send to chat" hand-off between tabs

This is the differentiator vs. a generic SQL editor: the agent is one click away, sharing the same schema grounding.

08 · Security & guardrails

This feature turns arbitrary user-typed SQL into a first-class input — previously only the agent (or its stub) authored SQL. The existing defenses hold, but one of them deserves hardening now that humans are at the keyboard.

Holds as-is
  • Read-only role — runs as agent_ro; no INSERT/UPDATE/DDL privilege even if a query slips the regex
  • RLS — SET LOCAL app.current_user_id scopes rows; user2 gets 0 rows
  • Statement timeout + row cap — sql_statement_timeout_ms, max_rows bound cost against the ~3M-row staging tables
  • Single statement — validate_select rejects ;-chained SQL
Harden now
  • Parser-based validation — validate_select is regex-based (keyword denylist). With direct user SQL, add an AST check (sqlglot/pglast): assert exactly one top-level SELECT, no CTE-hidden DML. Keeps the read-only role as the real backstop, regex/AST as defense-in-depth.
  • Per-user rate / cost limit — cap runs per minute + daily row budget (a runaway loop of heavy queries is the new abuse vector). Aligns with the "LLM cost drift" risk already tracked.
  • Access gating — decide who sees the tab (all users vs analysts+). See §11.

09 · Audit & the one migration question

We want every editor run audited exactly like an agent run, so it shows up in the admin dashboard. It fits app.query_runs cleanly — conversation_id, message_id, and dataset_id are all nullable — with one snag: query_runs.question is NOT NULL (db/init/01_schema.sql:94), and an editor run has no natural-language question.

Option A — tiny migration recommended

Add 0005_query_run_source: make question nullable and add source text NOT NULL DEFAULT 'agent' (values agent / sql_editor). Cleanest audit + lets the admin dashboard filter by origin.

Option B — no migration

Store question = '' (or the SQL's first line) and set engine = 'sql_editor' to distinguish. Zero schema change; slightly hackier audit semantics.

No other schema changes. events takes new free-text types (sql_editor_view, sql_query_submitted, sql_query_succeeded/_failed) with no migration — event_type is text.
