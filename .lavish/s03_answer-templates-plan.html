Teach the agent to produce insight reports — and let user feedback become its curriculum

Give the agent a versioned markdown knowledge tree it searches with tool calls — an "Insight Playbook" holding its know-how: how to build trend charts, what each domain's data means, what makes an insight useful. The agent plans how to present before it queries and returns a structured insight report: headlines (including related metrics), insights with comparison charts, a profile section, and numbered query references. The crux: users annotate feedback directly on report elements — each capture carries what it pointed at, a snapshot of it, and the knowledge version — so feedback scales into evaluations, evaluations drive knowledge-tree updates, and stale evals are detected and retired as the system evolves.

Direct answer The system has one flywheel: element-anchored user feedback → triaged into evals (or user memory) → eval runs grade the agent → a curator turns failures into knowledge-tree PRs → merged knowledge improves the next report.

Today the agent's know-how is one ~120-line f-string (services/data-agent/agent/llm_agent.py:42-165) and there's no feedback capture at all. This plan replaces the prompt with an AI-authored knowledge/ tree the agent greps via tools (§02), a typed report contract rendered as a data-insights page (§04–05), and — the centre of gravity — a feedback system where every thumbs-up/down is a durable, versioned, reviewable evaluation input (§06–07). Knowledge is plain markdown in the repo, so the runtime agent and your coding agent work from the same files.

02 · The Insight Playbook — a knowledge tree the agent greps

A new services/data-agent/knowledge/ folder. Retrieval = tool calls, the way a coding agent greps a repo

Per your annotation — yes, grep. The agent gets two knowledge tools and drives its own retrieval mid-run:

  • search_knowledge(query) — ripgrep-style full-text search over knowledge/, returning page names + matching snippets (with frontmatter applies_to terms boosting the ranking). Exactly how Claude Code finds things in a codebase.
  • read_knowledge(name) — load a full page into context.

INDEX.md stays pinned in the system prompt as the map (~1 line per page), so the agent knows what exists without loading it; the planning step (§03) is instructed to search before it plans. pgvector similarity (machinery already exists for user_memories, agent/memory.py) is a drop-in ranking upgrade later if grep recall proves too literal — same tool signature, so nothing else changes.

One source of truth, two consumers: the runtime agent greps these pages to answer; your coding agent edits the same files when you say "teach it X".

Analytics as tools, not templates

A small agent/analytics.py toolbox (rolling_average, growth_rate, latest_reliable) exposed as agent tools. Knowledge pages instruct the agent to use them ("compute growth with the growth_rate tool, never in your head") — headline math is code-computed, the agent stays free to decide what to compute.

Typed output via Pydantic AI

The agent's output_type becomes an InsightReport model (§04) — validated, retried on schema miss, so report structure is guaranteed, not requested. Free prose remains the fallback for non-analytical questions.

Every query becomes a reference

All successful run_sql calls are kept and numbered (Q1, Q2…) — replacing today's "only the last call survives" behavior (llm_agent.py:129-137). Insights cite the query that backs them; users copy any into the SQL editor.

04 · The InsightReport contract — every element is a feedback anchor

One typed shape flows agent → backend-api (app/routers/ask.py's AskResponse gains a report field) → frontend. Every headline, insight, profile, chart and query carries a stable element_id — that's what feedback attaches to (§06). Charts stay server-spliced Vega-Lite (the model never supplies data values; chart.py just learns layer + restricted interactive params).

06 · Feedback capture — the crux: annotations that scale into evaluations

The interaction is click-to-annotate: the report renders clean, the user clicks the element they want to judge, it highlights, and one feedback box opens against it — sentiment (👍/👎), numeric accuracy true/false, an optional ! flag for questionable figures, free-text feedback, and submit/cancel (§05 mock). After submit, the element shows a comment icon; questionable-number feedback shows the ! marker. A capture is only useful if, months later, you can still tell what it judged — so every submission records the element it anchored to, a snapshot of that element's content at the time, the rendered element HTML, a full report snapshot, client context, the knowledge-tree version and pages that produced it, alongside the run's trace (already persisted in query_runs.trace, migration 0007). That makes feedback durable enough to become an eval — and gives the curator/LLM the exact rendered context the user was responding to.

Triage: knowledge eval vs user memory

Per your direction, the default is everything is a knowledge eval (improves the agent for everyone). The admin review (§07) can reclassify a capture as user_memory when it's clearly a personal preference ("I only care about units") — those flow into the existing pgvector user_memories instead, injected per-user into the system prompt as today. The scope column exists from day one so triage can get smarter later without a schema change.

Promotion: admin batch → DB-backed eval cases

Per your decision: the admin runs a batch promotion from the review panel — select accumulated feedback, promote in one action. Promoted cases live in a new app.eval_cases table (question, distilled expectation, referent snapshot, knowledge version, status) — the database is the source of truth, so cases are toggleable in the admin UI; the eval runner pulls active cases at run time (a YAML export exists only for CI artifacts). Auto-actioning of evals is explicitly deferred — capture and promotion first.

Staleness: flagged, DB-toggleable, auto-archived

Per your decision: each eval run compares the current report's corresponding element against the stored snapshot; materially changed → the case's status flips to stale in app.eval_cases and it stops grading. The admin panel can toggle stale ↔ active at any time (re-affirm a still-valid expectation, or retire it); a case left stale for 3 consecutive eval cycles (stale_cycles counter) is auto-archived. Stale evals never silently fail the build.

07 · The learning loop — evals update the knowledge base, with you in the loop

To confirm your question directly: yes — the output of an eval run is (draft) updates to the knowledge base. Failures and low judge scores go to a curator session that diagnoses which knowledge page (or gap) caused them and drafts the markdown edit; the edit ships as a PR, evals re-run against it, you merge. Evals never rewrite knowledge silently — they generate the update, review applies it. Knowledge is git-versioned markdown, so every learning is diffable, attributable, revertable.

Admin review — feedback beside the rendered report

A new admin view (beside Datasets/Users/Query Runs in App.tsx's AdminDashboard, ~line 266): a feedback queue where each item re-renders the original insight report with the feedback pinned to the element it anchored to — like reviewing comments on a doc. The queue shows sentiment, accuracy/questionable-number flags, free-text feedback, the stored rendered element HTML, the report snapshot, filters for status/rating, and the user/question context. Triage runs in batch mode: select items, then Promote selected to evals · Save as user memory · Dismiss; a second tab lists app.eval_cases with the stale ↔ active toggle. Filters for status and rating. This is where you "see evals feedback and the rendered insights back" — the operational heart of the loop.

Wiki hygiene — the Karpathy lint pass

A periodic curator variant lints the tree: broken cross-links, contradicting pages, pages never loaded by any question in the last N runs (merge/retire candidates), index descriptions drifted from their body. Keeps the playbook from decaying into a junk drawer as it grows — and it's also what keeps the domains/ folders honest as new datasets arrive.
