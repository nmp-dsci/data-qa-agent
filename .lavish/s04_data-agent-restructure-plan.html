Sandbox-centric data agent: cheap model runs smart-model-authored skills

The runtime LLM pulls a bounded, pre-aggregated extract from the marts, then does the heavy lifting — growth, trends, insights, charts — as code in a locked-down sandbox, leaning on a library of skills authored offline by a smart model (Opus / Fable). A cheap runtime model stays cheap, but produces smart-model-quality analysis because the expertise is baked into the skills it calls.

Get smart-model-quality reports out of a cheap, few-turn runtime loop. Instead of the model orchestrating many fine-grained tools across ~19 turns (interpreting prose knowledge as it goes), it does one governed data extract and hands the analysis to code running in a sandbox — code it writes with the help of skills a smarter model wrote and tested ahead of time.

03

Why it strains

Two design choices that fight the model's tendencies.

Procedure encoded as prose

Pages like growth-rates.md describe how to compute in English. The model must find, read, interpret, and hand-carry each step through tools — the least reliable way to make an LLM follow a procedure, and the direct cause of the over-exploration in the trace.

The LLM is the execution engine

You already trust code over the model for the hard part — compute_trend exists because the prompt says “NEVER do arithmetic yourself.” The problem is the model still stitches those tools together one turn at a time. Move the stitching into sandbox code.

Your requirement — the model must flag skill gaps (we start from 0) Because the library starts empty, on day one almost every question hits a gap. So run_analysis returns a typed SkillGap whenever the model reaches for math no skill covers — the model may fall back to inline pandas to still answer, but it must name the gap. # run_analysis result carries an explicit skill-gap signal — never silent
class AnalysisResult(BaseModel):
    report: ReportDraft | None
    skills_used: list[str]         # ← logged per run → app.query_runs; the unit of eval/diagnosis
    skill_gaps: list[SkillGap]        # e.g. SkillGap(need="seasonality_adjust", why="no skill for it")
    used_inline_math: bool          # True ⇒ a skill probably should exist for this

Your two questions, answered

How skills plug into Pydantic AI · what language the sandbox should use.

Q · Pydantic AI

How do skills get used?

Skills live at two levels, and only a thin surface touches the agent:

  • Inside the sandbox — skills are a tested Python package the sandbox preloads; the model's generated code just import skills and calls skills.growth_rate(df, …).
  • To the agent — one Pydantic AI tool, run_analysis(code), executes that code. The model learns which skills exist from the system prompt (their signatures + smart-model-written docstrings/examples) or a small list_skills() / read_skill() tool.
  • Focus — Pydantic AI toolsets / prepare can expose only the skills relevant to a question, so the cheap model isn't drowned in options.
So in Pydantic AI it's one tool + a preloaded library, not 8 tools the model orchestrates. The agent decides what; skills carry the how.
Q · Language

pandas, or something faster?

Keep the model's surface in Python + pandas. The bottleneck here is model correctness and turns, not runtime speed:

  • LLMs write pandas most reliably — a cheap model is far more accurate in Python than in Polars / DuckDB SQL / Rust. Wrong code = more turns, the exact cost we're cutting.
  • Volume is handled upstream — the extract is filtered / pre-aggregated in SQL, so the frame is already small. pandas speed is a non-issue.
  • Hide fast engines in skills — if one skill is genuinely heavy, implement it in Polars / DuckDB / numpy under the hood. The model still just calls skills.foo(df).
Python + pandas at the surface; fast engines hidden inside skills; heavy aggregation pushed to SQL. Don't trade a non-bottleneck (speed) for a real one (model errors).

The sandbox — how it's built, and how it scales

How agents usually run LLM-written code, why you picked Pyodide/WASM, and what deploy looks like.

How teams usually sandbox LLM-written code (2026)

The field has converged on four isolation levels. The rule of thumb: shared-kernel containers are fine for code you wrote, not for code an LLM wrote — a single kernel bug is an escape. Stronger isolation costs either overhead or library reach.

ApproachIsolationLibrariesCold startWho uses itFit here
Container (Docker / runc)Shared kernel — weak for untrustedAnyFastDaytona (default)Too weak for LLM code
gVisor (user-space kernel)Syscalls re-implementedMostMediumModal, GoogleSolid, ops overhead
Firecracker micro-VMOwn kernel per run (KVM)Any~100 msE2B, AWS LambdaGold standard, VM fleet to run
Pyodide / WASM ✔No syscalls at all — capability-scopedWASM-compatible onlyms, in-processNVIDIA (client-side)Best fit — see below

Research sources: Modal — best code-execution sandboxes · Northflank · Sandboxing guide — Firecracker / gVisor · Spheron — E2B / Daytona / Firecracker

Why Pyodide/WASM fits this app

  • We need almost nothing exotic — pandas + numpy over a small in-memory frame. No GPU, no network, no big-data engine. WASM's library limits mostly don't bite.
  • Strongest isolation, least ops — no syscalls means no kernel to escape; no VM fleet or container orchestrator to run. Ideal for a small/solo deploy.
  • The frame is bounded upstream — extract-time SQL filters / pre-aggregates, so per-run memory is tiny and predictable.

The one constraint to respect

Pyodide runs CPython compiled to WebAssembly — pandas and numpy ship as WASM wheels, but arbitrary C-extension libraries may not. So the “hide a fast engine (Polars / DuckDB) inside a skill” idea has a caveat: use its wasm build if one exists, otherwise push that heavy aggregation to SQL at extract time instead. For v1 the skills need only pandas+numpy, so this doesn't block us.

How it's implemented & the prototype→hardened path

boundaryThe agent only ever sees one tool — run_analysis(code). The executor behind it can swap without touching the agent or the skills. That's what lets us prototype fast, then harden.
Phase AQuick sandbox to prove the win — a locked-down subprocess (no net / no fs / no env, CPU + memory + wall-clock caps) running the skills over the injected frame. Fast to stand up; measures the turns/token drop immediately.
Phase BHarden to Pyodide/WASM — swap the executor to run the same code inside a Pyodide runtime (hosted via a WASM engine / worker). Same run_analysis surface, same skills, strictly stronger isolation.
How it scales on deploy WASM instances are cheap, stateless, and per-request: spin one up in milliseconds, cap it, discard it. No warm VM pool to manage. Horizontal scale is just more app workers, each holding a pool of pre-initialized Pyodide runtimes. Because the extract is bounded, per-run memory stays small and predictable — the opposite of a Firecracker fleet that needs its own orchestration layer.

Risks & the sandbox security bar

This is the cost of Option B — treat it as a build requirement, not a caveat.

The sandbox must, before it runs a single line of model-written code:

✕ network ✕ filesystem ✕ DB credentials ✕ env / secrets ⏱ wall-clock cap ▣ memory cap ◱ CPU cap ＝ only the extracted DataFrame
High

Sandbox escape / resource exhaustion

Model-written code is untrusted. A weak sandbox is an RCE path. Decided: Pyodide/WASM for the hardened runtime (no syscalls to escape); a locked-down subprocess only for the Phase A prototype. Trade-off accepted — skills stay pandas+numpy so WASM's library limits don't bite (see §07).

High

Data governance boundary

RLS is enforced at extract time; the sandbox must receive only that scoped frame — never a live connection or a way to re-query. Enforce via a frozen DataFrame passed by value + zero DB handle in scope.

Medium

Cheap model writes wrong code

Free-form code can be incorrect. Mitigation: skills do the risky math (tested once), the model mostly wires them together; run_analysis returns errors so it can self-correct within a turn budget. Cap retries so a broken run can't loop.

Medium

Chart / report parity

Skills must reproduce today's make_chart specs and ReportDraft shape so the frontend renders identically. Snapshot-test skill output against current reports in the eval set.

Low

Interim: the token ceiling resolved

Until this lands, the heavy question falls to stub. Decided: bump agent_total_tokens_limit now as a stopgap (tokens are ~6× cache-inflated) — Phase 0. The restructure then makes the ceiling moot.
