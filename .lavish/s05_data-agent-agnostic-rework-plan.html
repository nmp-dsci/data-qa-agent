data-qa-agent · s05 · agent rework

From “NSW property analyst” to a dataset-agnostic insight agent

The sandbox+skills restructure (s04) already gave the agent a clean shape: one governed SQL extract, then analysis as code that calls tested skills. This plan removes the last thing hard-wired to one dataset — the property-specific system prompt and schema — and turns domain knowledge, decision tracing, and visualisation into first-class, evaluable surfaces. The compute model doesn’t change; what the agent knows and what it exposes does.

02

The rework, mapped to what you asked

Your five asks collapse into five workstreams. Each is scoped so it ships independently.

W1

Dataset-agnostic agent

“make it agnostic to dataset … not just a data-analyst for NSW property”

Rewrite the system prompt to a neutral insight-agent role. Replace the pinned full schema with a lean mart index (table name + one-line purpose). Add an explicit “no answer” outcome so the agent can say a dataset doesn’t cover the question instead of forcing a report.

W2

Marts + knowledge, not a wall of prompt

“tell it to use tables in marts schema … bring context in with search instead of a massive system prompt”

The agent discovers marts at runtime and pulls domain grounding from the knowledge base (the property JOIN_HINT moves out of code and into per-dataset pages). Knowledge becomes the lightweight semantic contract — the industry’s highest-leverage accuracy move.

W3

Every decision, per run_id

“show for each run_id all the decisions the LLM made … including skills, for diagnostics/evaluation”

Promote the dropped deps.steps into the persisted trace, add a one-line rationale to each pick (table, knowledge page, skill, chart), and render an ordered Decision Log in the admin run view. The same log powers trajectory evals.

W4

Python / codegen skills

“have skills for python, code gen for sandbox”

Largely already built (s04). This workstream is a de-couple pass: rename property-specific skills/args to generic ones (group_col, not “suburb”), and split the catalog into generic analysis vs domain skills so a new dataset reuses the core.

W5

Visualisation skill library + “when to use what”

“improve visualisation … skills for time-series / bar with secondary y-axis as line … guide the LLM when to use”

Add the charts you named — most importantly a dual-axis combo (bars + a secondary-axis line, e.g. volume vs price) — plus distribution and part-to-whole. Then add a chart-selection guide (a knowledge page + sharpened skill docstrings) that maps question shape → chart, so the model stops defaulting to a line chart. Charts stay server-validated Vega-Lite; the frontend renderer already handles any spec.

Your annotation — does this handle joining multiple tables?

Yes — two ways, both already built in the s04 sandbox path. The single linear box in the earlier diagram hid it; extract is callable up to max_sql_attempts (8) times and each result is a named DataFrame handed to the sandbox.

  • Join in SQL — one extract whose SELECT JOINs marts (e.g. sales ⋈ rent on postcode·property_type·month). RLS is enforced per-table, so the join stays governed. Best when the join is simple and you want the DB to do it.
  • Join in pandas — extract(sql, name="sales") then extract(sql, name="rent"); both frames land in the sandbox and a skill merges them. gross_yield(rent_df, price_df, key_cols=…) is literally the two-frame case — it exists today. Best when each side needs its own smoothing/aggregation first (the yield case).

Which strategy the agent picked — and the tables/keys — is captured in the Decision Log (W3), so a wrong join is visible in diagnostics and assertable in evals.

W2 · Knowledge base as the semantic contract

The knowledge tree already has the right shape. The change is to make it the single home for domain grounding and to add one semantic-profile page per dataset — the thing a semantic layer gives you: grain, measures (how to compute the metric from raw columns), dimensions, join keys, and gotchas. This is the highest-leverage accuracy move in the plan (see §06).

Semantic-profile fieldExample (property_rent)Why the agent needs it
grainpostcode · property_type · bedroom_band · monthStops SUM-across-grain errors
measuresavg rent = total_weekly_rent / n_rented (additive parts)Never sum a median; compose rates correctly
join keysrent ⋈ sales on (postcode, property_type, month) — not suburbRent has no suburb dimension
gotchaskeep thin months; resolve suburb→postcode via bridge firstThe property-specific traps, out of code

Adding a dataset becomes: drop CSV → dbt mart → copy _template/ → fill one semantic page. No agent code changes. The applies_to frontmatter already auto-registers it for retrieval.

Your annotation — is there data-agnostic knowledge? Yes — and the tree should make it a first-class axis.

Knowledge should split on two axes, not one:

  • domains/<slug>/ — data-specific. Grain, measures, join keys, gotchas for one dataset. (Exists today.)
  • analysis/ — data-agnostic (new). How-tos that hold for any dataset: smoothing volatile series, choosing a growth window, when a median beats a mean, the chart-selection guide (W5), safe-SQL idioms.

Same three-tier retrieval; agnostic pages stay eligible regardless of domain, so analytical know-how is written once and reused across every dataset — and a new dataset inherits it for free.

Your annotation — the agent must pick (and join) tables Yield is the perfect example.

Property yield = annual rent ÷ sale price, on a 6–12 month moving average to kill volatility. That spans two marts:

# one logged decision, captured in full:
tables : marts.property_sales + marts.property_rent
join   : (postcode, property_type, month)
measure: gross_yield, 12-mo smoothed

Table selection is itself a logged decision (W3) — the agent may pick 1..N marts. The measure recipe (“yield = rent/price, smooth 12m”) lives as an agnostic analysis/ page + a tested skill (gross_yield exists — add the smoothing window). The model composes the recipe; it never invents the formula.

Decided — v1 uses keyword search You chose markdown semantic pages with keyword search for v1. The pgvector-embedding upgrade (reuse the memory stack so “what drives rents up” matches by meaning, not literal words) stays an easy, isolated follow-up — no interface change — if retrieval ever proves too literal.

W3 · The Decision Log — per run_id, for diagnostics & eval

This is the fix for “I can’t see what the agent picked.” The signal already exists — it’s just discarded. In answer_with_sandbox() the returned trace is rebuilt from model messages and deps.steps is never merged in, so the tagged picks (which knowledge page, which table, which skill) don’t reach query_runs.trace.

# sandbox_agent.py — answer_with_sandbox(), today
trace = _build_trace(messages)          # model turns + tool calls only
trace.append({"kind": "analysis", ...})  # one summary line
# deps.steps  ← the tagged decisions are dropped here

# target: fold the tagged picks in, each with a one-line 'why'
decisions = _decision_log(deps.steps)   # ordered: knowledge / table / sql / skill / chart
trace = _merge(_build_trace(messages), decisions)

Then two additions turn a step list into an evaluable trajectory:

  • Rationale on each pick (decided: why= param) — you chose to have the discovery tools take a short why= so the model states intent per pick (which table, page, skill, chart) rather than us inferring it. Best diagnostics; costs a few tokens/turn, bounded by the existing caps.
  • An ordered Decision Log in the admin run view — “① knowledge: property-rent-overview · ② table: marts.property_rent · ③ skill: trend_series · ④ chart: dual_axis”, each with its one-line why. Extends the existing AgentTrace component.
  • Trajectory evals — evals/journeys.yaml can now assert on the path, not just the answer: expect mart = property_rent, skill used = trend_series, chart = dual_axis. Exactly the “efficiency / hallucination / adaptivity” trajectory framing the field has moved to (§06).
Your question — new table, or extend query_runs? Extend query_runs.trace (jsonb) — don’t add a table yet. The trace already lives there, one row per run, RLS-scoped; the Decision Log is just a better-structured slice of it. A separate normalised decisions table only earns its keep once you want cross-run analytics at scale (“how often did it pick the wrong mart?”) — at which point a materialised view over the jsonb, or an ETL into an eval table, is the right move. Start in jsonb; promote later if eval dashboards demand it.

Mirrors how the field is packaging viz as an agent skill that turns “data into a visual language” (AntV chart-visualization-skills) rather than leaving chart choice to the model’s guess.

Your question — is Vega-Lite the right library, or move to something richer?

Keep Vega-Lite as the governed default; add a second renderer only for what it genuinely can’t do. The reason it fits here is the governance model, not habit: the model emits a spec, the server validates it against an allowlist and splices in real governed rows, so the model can’t fabricate chart numbers. An imperative library (D3) or a free-form codegen chart would throw that away.

  • Richer within Vega-Lite — it already does layering, dual-axis, faceting, interaction; drop to full Vega for the rare advanced spec. Covers ~90% of “rich”.
  • Add ECharts as a 2nd skill backend only for the types Vega-Lite does poorly — geospatial maps, sankey/network, gauges. Same skills.* interface, same server-splice-the-data contract, so the library stays swappable and the governance holds.
  • Avoid a model-writes-arbitrary-JS renderer — it breaks the “can’t fabricate numbers” guarantee that makes this agent trustworthy.

The agent emits the signal today; this closes the loop. The gate is deliberate — candidates are reviewed and tested before they change what the agent relies on.

① Runtime sweep (your idea — yes)

A scheduled backend job reads the signals the agent already writes to query_runs: skill_gap, used_inline_math, no_answer, repeated retries, thumbs-down. It clusters them (“12 questions wanted a yield the agent hand-rolled”) and drafts a candidate skill or page. Output is a review queue, never an auto-merge.

② Build-time hook (your idea — yes)

Exactly the DB/pipeline hook you suggested: a dbt post-hook (or migration step) fires when a mart’s shape changes, diffs the live schema against each semantic page, and flags drift. knowledge_version() — a content hash already recorded on every report — is the primitive; the hook just turns “this report used stale knowledge” into a task.

③ Human/agent-in-the-loop

Candidates are reviewed (Opus-authored per the s04 restructure decision), skills carry tests, and the trajectory evals (W3) catch regressions before merge. Merging bumps knowledge_version, so you can always tell which knowledge produced which past answer.

Why not just auto-write skills? Skills do the risky maths and are trusted precisely because they’re tested once and reused. An auto-merged skill would erode that guarantee. The sweep’s job is to surface and draft — turning scattered gap flags into a prioritised, reviewable backlog — not to silently expand what the agent relies on.
06

How the field builds insight agents (and where this lands you)

Three patterns from 2025–26 practice; each maps to a workstream above.

Semantic layer > raw text-to-SQL

The consensus reliability move: define grain, measures, joins and policies once and let the agent reason over that, not raw tables. Paired benchmarks show large jumps — e.g. one frontier model going from 90% → 98% accuracy with a modeled semantic layer vs raw text-to-SQL.

→ W2 Your dbt marts + per-dataset semantic pages are a lightweight semantic layer.

Evaluate the trajectory, not just the answer

Final-answer accuracy hides how the agent got there. The field now scores the reasoning trace — tool choice, efficiency, hallucination, loop detection — walking from output back to the exact span that broke.

→ W3 The Decision Log makes each run a scoreable trajectory; you already have the loop-guard and token caps that trace framing recommends.

Visualisation as a guided skill

Mature tools don’t just plot — they pick the statistically sound chart for the data’s shape and add analytical overlays that highlight the key period. Chart choice is packaged as a reusable agent skill, not left to the LLM.

→ W5 The selection guide + dual-axis/overlay skills close this gap.

08

Risks & the decisions you locked

What could bite, and the choices this review settled.

MED

Pull-on-demand doesn’t shrink a single run’s context — it caps how context scales

Fair catch from your review: once the agent searches and loads the property-sales page, that schema does sit in every later turn of that run — so one question isn’t necessarily cheaper. The win is different. Today’s pinned block carries every dataset every turn, so prompt cost grows O(datasets). Pull-on-demand makes a run carry only the 1–2 datasets it actually touches: a rent question never pays for sales schema, and adding dataset #10 doesn’t inflate the other nine. The lean mart index (names only) stays always-on so discovery still works; mandatory search_knowledge keeps grounding reliable; trajectory evals guard against a retrieval miss.

LOW

Cheaper model (DeepSeek) may under-use optional tools

A weaker model given more discovery tools can loop or skip search_knowledge. The existing request/token caps and loop-guard bound the blast radius; the Decision Log will make any regression visible immediately.

LOW

Dual-axis charts can mislead

Two independent y-scales invite spurious “correlation.” Encode it as a deliberate skill (fixed styling, explicit axis titles) and note in the selection guide when not to use it — same house-style discipline as the other chart skills.

Locked · 1

Semantic pages in the knowledge tree, token-overlap retrieval. pgvector stays a drop-in upgrade if it proves too literal (W2).
