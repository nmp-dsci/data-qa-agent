From fixed report renderer to a page-based report engine

The direction is locked: the agent emits pages of governed objects (data + intent — never chart specs or layout), the frontend owns a fixed template registry and renders everything with visx. Every answer leads with a Summary page (latest number + growth) and explains itself with an Insights page (driver analysis). Around that core: de-monolith the React app, refresh the design system, and reshape the tabs.

React 19 + Vite + TS App.tsx 1,668 lines → feature modules Charts: visx only, frontend-authored Answer = Summary → Insights pages 3 tabs → 4 (+Settings)

03

The agent output — exact JSON contract

The agent returns an ordered list of pages; each page names a template and fills its regions with typed objects. Each object keeps a stable element_id. The schema is validated before rendering.

{
  "answer_id": "run_9f2a…",
  "pages": [
    {
      "template": "summary",          // picked from the published registry
      "objects": [
        { "type": "kpi",   "element_id": "k1", "region": "hero",
          "data": { "label": "2br unit median rent", "latest": 671, "unit": "$/wk",
                   "growth": { "yoy": 0.061, "mom": 0.008 },
                   "series": [ {"month":"2025-01","value":632}, … ] } },
        { "type": "trend", "element_id": "t1", "region": "chart",
          "data": { "intent": "line", "x": "month", "y": "value", "series": [ … ] } }
      ]
    },
    {
      "template": "insights",
      "objects": [
        { "type": "breakdown", "element_id": "b1", "region": "chart", "explains": "k1",
          "data": { "intent": "bar", "dimension": "bedroom_band", "measure": "median_weekly_rent",
                   "rows": [ {"bedroom_band":"2","value":671,"growth":0.061}, … ] } },
        { "type": "insight", "element_id": "i1", "region": "note",
          "data": { "text": "3-bed units drove the rise (+7.3% YoY)", "refs": ["b1"] } }
      ]
    }
  ]
}

template = published registry id · region = slot in that template · type+data = governed content (visx renders from data.intent) · explains/refs wire insights to the numbers they explain.

  • Schema validation on output (Pydantic) — reject/repair malformed pages before they reach the frontend
  • Tracing update — record object-build → template-pick → page-compose as steps in the query_runs trace (visible in Admin + inspect_run.py)
  • Retire charts.py spec skills + validator; build_report becomes build_pages

Pages & templates — Summary always, Insights when it explains

The answer is an ordered list of pages over the same governed objects. Page 1 · Summary always leads — it captures the answer and builds trust: the latest number in a time series with its growth rates. Page 2 · Insights explains it — driver analysis over the attributes that roll up to the top-line. A detail/report page is not required.

Drivers by attribute (property type · bedrooms) that explain the top-line

Insights = driver analysis (a real agent capability, not just a breakdown)
The Insights page identifies the biggest discriminator: which available attribute most explains high vs low values of the headline metric — e.g. for rent, is property_type or bedroom_band the stronger driver of higher rent? For a suburb / comparison, it surfaces mix differences — the % split of property type / bedrooms that explains the gap (suburb A skews 3-bed houses; suburb B skews 1–2-bed units). Implemented as an agent driver / variance-decomposition skill that ranks attributes by explanatory power, then renders the top drivers into the insights template. Ranking method (variance decomposition vs mutual info vs % contribution) is an open question below.

Published template registry — the agent picks by id, fills regions

Fixed registry, not agent-directed free-form layout (decided, revisitable in §11): predictable, themable, testable — and every region is a known slot for feedback/evals. New template = new frontend layout + registry entry.

05

Chart layer — visx only, frontend-authored

Decided: the frontend authors every chart with visx (D3 primitives) for maximum control and one consistent house style. Vega-Lite (agent-authored) is retired; Recharts was considered and dropped — one library, one styling system.

Chart functions to build (the visx library)

FunctionIntentUsed for
TrendlineSummary time series; multi-series by attribute
BreakdownbarInsights drivers; metric by dimension (+growth labels)
Comparegrouped-barhouse vs unit, suburb vs suburb
KPITilekpiLatest number + growth + sparkline
Axis · Legend · Tooltip—Shared primitives all charts compose

~4–5 reusable components, reused by chat answers, the SQL editor's "result → chart", and the Admin KPI strip.

What visx buys · what it costs

  • Compose tiles + charts + annotations in one layout the template controls
  • Custom marks — growth callouts, reference lines, gradients, lollipops
  • Interaction when useful: hover, brush, crossfilter, animation
  • One styling system in code → every chart matches the house tokens
  • Low-level: scales/axes/legends are ours to build (hence the shared primitives)
  • Migration: agent stops emitting specs; VegaChart.tsx + vega deps removed

10

Phased rollout

Incremental — the app keeps running and each phase ships something usable. Phase 3, the report engine, is the centerpiece and the biggest lift.

Phase 1foundation

Tokens + routing + query layer

Tokenize styles.css, add react-router (tabs → routes), add react-query. No visible feature change — sets up everything else.

Phase 2de-monolith

Extract features from App.tsx

Split Chat, Admin, Report* into features/* + ui/* primitives (shared DataTable, Card, Tabs). Behaviour identical, code testable. Feedback-popover logic preserved.

Phase 3report engine

Templates + visx charts + agent pages contract

Frontend: template registry (summary, insights, one/two-col), object renderers (type→component by region), visx chart functions (Trend, Breakdown, Compare, KPITile, Axis/Legend/Tooltip). Agent: system prompt + few-shot to emit pages/objects with schema validation; driver-analysis skill for Insights; tracing records object-build → template-pick → page-compose; retire charts.py + validator. Admin: agent_config table + Agent-Config view, demo-seeded with the Hornsby example.

Phase 4tabs & panels

Chat history sidebar → SQL catalog → Settings

History sidebar is nearly free (conversations already in query_runs); catalog grows out of the schema tree; Settings is light. (Dashboards tab deferred — out of scope.)

Phase 5cross-cutting

Streaming, ⌘K, export, mobile, a11y

Layer the product-wide features once the structure is in place.

Decision log, risks & what's still open

Decision log — locked during this review

Charts
visx only, frontend-authored. Agent stops emitting chart specs (retire charts.py + validator + VegaChart.tsx + vega deps). Recharts considered and dropped — one library, one style.
Agent output
Pages of governed objects — pages[] → {template, objects:[{type, element_id, region, data+intent}]}. Data + intent only; schema-validated; element_ids preserved for feedback/evals.
Layout freedom
Fixed template registry (summary · insights · one-col · two-col). Agent picks a published id + fills regions; it cannot invent layout.
Pages
Summary always leads (latest number in a time series + growth — builds trust); Insights explains (driver analysis: biggest discriminator, mix differences). Detail/report page dropped.
Tabs
3 → 4: Chat, SQL Editor, Admin, +Settings. Dashboards tab dropped (dashboard = inline view-page). History = Chat sidebar (new conversation = new thread; multi-turn within). Catalog = inside the SQL editor.
Refactor
Incremental — one feature per PR, app runs the whole time.
Evidence
Worked example pinned to real data — ai_plans/s07_ui-refresh-plan/ (Hornsby/2077, marts.property_rent).

Risks / migration concerns

  • Report-engine scope — Phase 3 spans agent prompt, output schema, tracing, frontend registry and visx; keep it behind the old renderer until pages parse reliably
  • LLM output validity — pages/objects JSON needs strict validation + repair; eval cases should cover malformed output
  • visx build-out — cap the chart library at ~4–5 reusable components; resist bespoke one-offs early
  • Feedback pinning — element-pinned popover must survive both the extraction and the new page renderer
