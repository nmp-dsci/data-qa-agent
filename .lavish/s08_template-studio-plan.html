1 · Goal & why now

The point of the preview is to show how the Data-Agent informs the frontend to build a page, then watch that page get filled with insights that answer the question. Today the preview is a hard-coded demo that quietly diverges from what actually renders in chat — so it can't be trusted as a test surface, and it hides the contract it's supposed to teach.

Trust

The preview must render through the same code path as a real answer, or it proves nothing.

Legibility

Every preview should expose the JSON the agent would have emitted to produce it.

Coverage

Every layout × every chart-per-column must be checkable — by a human and in CI.

5 · Proposed approach

Five phases. Phase 1 removes the fork so everything after it is tested once and rendered once.

1

Unify the renderer & the registry — foundation, unblocks the rest

  • Extract one <PageLayout> from PagesView that takes a page + template and places objects. Chat, preview, and the new playground all call it. Delete DemoPage.
  • Make registry.ts the single frontend source; assert at build/test time that its ids match TEMPLATE_IDS and the 0014 seed (a tiny contract test), closing Problem B.
  • Fixing one-col here fixes it in both chat and preview at once.
2

Rename & relabel — copy + one tab id

  • Tab Agent-Config → Template Studio; section Demo preview → Template Preview.
  • Rewrite the caption to describe the contract-teaching purpose, not "demo".
3

Contract inspector — the "show the JSON" ask

  • Under every Template Preview, a collapsible ▸ Contract JSON (what Data-Agent sent) showing the real Page object that produced the render.
  • Syntax-highlighted, copy button. It updates live with the preview.
4

Column model front-to-back: chat cut-over, three-col + the Template Playground — the core change

  • Ship the column model (D2): template = N ordered columns; placement is positional (columns[i][j]), meaning stays typed (type + optional semantic role replacing today's region — see §8). summary/insights become presets over it; add three-col + a 3-track CSS grid.
  • Data-agent update (yes, it's needed): rewrite compose_pages() in agent/pages.py to emit the columns contract, and update the report/chart skills + page-composition prompt so the model picks a Template Studio template and fills its columns. The Pydantic envelope validates the new shape only — the agent physically cannot emit layout the frontend can't render.
  • Chat runs on it (D4): /ask answers render through the same unified <PageLayout> + registry the Studio uses — Template Studio isn't a side demo, it's the literal catalogue of what chat answers are built from. Answer a question → the agent populates a template with insights → chat renders it.
  • Clean cut-over, no backward compat (your call): migration 0015 clears the stored pages/report payloads in app.query_runs (run metadata, SQL text and audit rows are kept). Old conversations reopen without a rendered report (re-ask to regenerate); in exchange we ship zero dual-shape validation or legacy-renderer code.
  • Per-object height (your note): each object may carry an optional data.height (px, or semantic sm/md/lg/fill). A lone chart in a column can be made taller — or fill — so it doesn't leave an empty gap beside a taller stacked column. The visx charts already accept a height prop (Trend/Bars default 280), so this is mostly plumbing the value through <PageLayout>.
  • A playground: pick layout one-col / two-col / three-col; for each column pick a chart type (tiles · line · bar · grouped-bar · kpi, extensible) and its height. It generates the contract JSON and renders it through <PageLayout> live.
  • This makes the layout control, "what can go in each column", and column balance visible and manipulable.
5

Combinatorial UI tests · Playwright — same interface a human uses

  • Drive the playground in a real browser: for every layout × per-column chart type, assert the page renders (chart SVG present, no error boundary, object count matches the JSON).
  • Interaction tests: switching layout re-flows objects; changing a column's chart type re-renders; the Contract JSON stays in sync with the DOM.
  • Live-chat E2E (D4): ask a real question in chat and assert the answer arrives as a column-model page that renders — proving the data-agent → contract → frontend path end-to-end, not just the playground.
  • Runs in CI headless against make up, mirroring the existing smoke test's spirit.

How an object lands in the layout — placement is positional, meaning is typed

Your question: "shouldn't the object say 'headline' / 'insights' for column 1 and 'chart' for column 2 — how do you know which object lands where?" In the column model those are two separate concerns, and splitting them is the point:

WHERE it lands — position in columns

columns[i][j] is the placement: object in array i renders in column i (left→right), slot j (top→bottom). The data-agent controls layout purely by where it puts the object in the JSON — no name lookup, nothing to mismatch, and the renderer can never disagree with the contract. (Today's named region is exactly such a lookup, and a typo'd or unknown region silently appends to the end — that ambiguity goes away.)

WHAT it is — type (+ role)

type: kpi | trend | breakdown | … picks the renderer. An optional role: "headline" | "insight" | … keeps the semantic label — carried over from today's region so element-pinned feedback, eval snapshots, and the agent's own reasoning ("this KPI is the headline") keep working. role never affects placement.

So "columns": [ [kpi, insight], [trend] ] reads: column 1 = headline tile then insight card; column 2 = the chart. Presets like summary are just named recipes that expand to a columns arrangement. Plus optional per-object height from your earlier note.
