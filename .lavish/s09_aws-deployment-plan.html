Plan · s09 data-qa-agent Re-platform: Azure → AWS Auth: Entra → Google

End-to-end AWS deployment + CI/CD

Take the working local stack (3 services + Postgres + 2 jobs, all in docker compose) and stand it up on AWS with a push-button CI/CD pipeline — swapping the Microsoft Entra login for Google sign-in along the way.

🎯
Definition of success
The app is fully deployed to AWS and I can open a public HTTPS URL, log in with Google, and use it exactly as I do locally — ask a question, get a governed answer + chart, and the admin dashboard works. The cloud smoke_test.py passes against the live URL.

✅ Already true today

  • Fully working local stack via make up
  • 3 containerised services + 2 one-shot jobs
  • Auth is runtime-switchable (/auth/config) — pluggable by design
  • Alembic migrations run identically local & cloud
  • CI (ci.yml): lint · tests · build · smoke

🔧 What this plan changes

  • Cloud target: Azure → AWS
  • Login: Entra/MSAL → Google OIDC
  • IaC: Bicep → Terraform (cloud-neutral)
  • Frontend: dev-server → static build on S3/CloudFront
  • New deploy-to-AWS GitHub Actions workflow

♻️ Untouched (portable app code)

  • The agent, RLS model, tools, memory
  • The dbt + dlt pipeline logic
  • FastAPI endpoints & the DB schema
  • The JWKS-validation + JIT-provisioning pattern (reused for Google)

Grounded in the repo: AGENTS.md already flags the portability path (Container Apps ↔ Fargate/App Runner · Flexible Server ↔ RDS · Entra ↔ Cognito · Key Vault ↔ Secrets Manager) and recommends Terraform "if you may move clouds." This plan executes that path.

3 · One question, end-to-end on AWS (with Google login)

The request path is unchanged from the repo's design — only the identity provider and the hosting swap.

sequenceDiagram autonumber participant B as Browser (CloudFront) participant G as Google (OIDC) participant A as backend-api (App Runner) participant D as data-agent (App Runner) participant P as RDS Postgres (RLS) B->>G: Sign in with Google G-->>B: ID token (RS256, JWKS-signed) B->>A: POST /ask (Bearer ID token) A->>A: Verify token vs Google JWKS (aud + iss) A->>P: JIT upsert user (by Google sub); SET LOCAL user_id A->>D: delegate question (schema + question) D->>P: run_sql (read-only role, SELECT-only, under RLS) P-->>D: rows the user is allowed to see D-->>A: typed answer + chart spec A-->>B: answer streamed to chat UI Note over A,D: every step traced in Logfire

🔑 The key insight: the backend already does exactly this for Entra in app/auth.py (EntraVerifier loads OIDC metadata → validates against JWKS with audience+issuer → _provision_entra_user JIT-upserts keyed by oid). Google is a standard OIDC provider, so this becomes a near-drop-in GoogleVerifier keyed by sub. See §4.

4 · Auth rework — Google login investigation

You asked to weigh Google email login against the current Azure Entra (MSA). Two ways to do Google on AWS. Both reuse the existing /auth/config switch and the JWKS-validation pattern, so neither is a rewrite.

A · Google OIDC — direct

Recommended

Frontend uses Google Identity Services; backend validates Google's ID token against accounts.google.com JWKS with your OAuth client ID as the audience. No extra AWS service.

Why it fits
  • Reuses the exact EntraVerifier shape → ~a day of backend work
  • Zero added infra, zero added cost
  • Frontend swaps MSAL for a Google button behind the same auth.ts interface
  • Cloud-neutral — not tied to AWS
Cost
  • Google-only IdP (fine for "log in with Google email")
  • You manage the OAuth consent screen & client yourself
  • No hosted UI / MFA broker (Google handles its own MFA)

B · Cognito + Google federation

Managed broker

An AWS Cognito User Pool brokers login, with Google configured as an identity provider. Backend validates Cognito-issued tokens.

Why you'd pick it
  • Managed hosted UI, MFA, password users, token rotation
  • Add more IdPs later (Apple, SAML) without frontend changes
  • "AWS-native" — one console for identity
Cost
  • Another service to provision + terraform
  • Cognito token shape/claims differ → more mapping work
  • Small per-MAU cost; some well-known Cognito quirks
💡
Recommendation: Option A (Google direct). The repo's auth is already a generic OIDC verifier + JIT-provisioning keyed by a stable subject claim. Google slots straight in for the least effort and no new infra. Keep Option B in your back pocket for when you want multiple IdPs or a managed MFA/hosted-UI broker. Either way the dev stub stays as the offline/local default.

5 · Compute choice — how the containers run on AWS

One real fork. Both keep the frontend on S3+CloudFront and Postgres on RDS; they differ in how the two API containers run.

ECS Fargate

Harden to later

Serverless containers with a real VPC. Backend behind a public ALB; data-agent private via Cloud Map. Jobs = ECS run-task.

  • ✅ True private networking for the agent + RDS (matches "no public ingress")
  • ✅ One-off tasks map the migrate/pipeline jobs cleanly
  • ✅ Closest analogue to the repo's Container Apps design
  • ⚠️ More Terraform (VPC, subnets, ALB, NAT); ~always-on min tasks

AWS App Runner

✓ Chosen for v1

Fully-managed container service straight from ECR. Scale-to-zero-ish, minimal networking.

  • ✅ Least infra to write; fastest to first deploy
  • ✅ Cheaper when idle (scales down)
  • ⚠️ Private service networking is newer/limited; agent isolation is weaker
  • ⚠️ Still need ECS/CodeBuild for the migrate + pipeline jobs
⚖️
Decided: App Runner. It's the lower floor (~$30–50/mo, see §6), scales down when idle, and gets you the real experience fastest. Because the app code and images are identical, graduating to the hardened ECS Fargate topology (private agent, ALB, WAF, custom domain) later is a Terraform change, not a rewrite. ECS Fargate remains the recommendation if you want the private-agent isolation from day one. Full cost + scaling breakdown in §6.

*
Why isn't App Runner compute basically free? App Runner keeps at least one instance warm per service and bills that provisioned memory 24/7 (~$0.007/GB-hr ≈ $2.50/mo per 0.5 GB service), then adds active vCPU only while a request runs (~$0.064/vCPU-hr). With two services that's a few dollars idle; the range climbs because the data-agent holds a request open while it waits on the LLM, so "active" time can be longer than you'd expect. Ways to push it toward the low end: consolidate to fewer services, use the smallest size, or accept cold starts. It's still the cheapest real-container option here.

† What "traffic" means here: 1 question = one /ask request that holds a connection open ~5–15s while the agent runs SQL and the LLM composes the answer. A user typically fires several questions per session, so "~50–500/day" is roughly a dozen to a few dozen active users; "~1k–5k/day" is a few hundred. "Peak concurrent" = questions in flight at the same moment — the number App Runner's ~100/instance limit and the RDS pool actually care about.

Cold-start reality: App Runner's minimum is 1 instance per service — it does not scale to $0 like Cloud Run, so the ~$5 warm baseline is your true floor even at zero traffic. Biggest lever for the absolute floor: co-locate backend-api + data-agent into one App Runner service (~$2.5 baseline) at the cost of clean separation. And remember: at every band above, the LLM bill dwarfs these infra numbers once the real agent is on.

How each layer scales with users

  • Frontend — a CDN; effectively free to scale. Never your problem.
  • Compute — autoscales on concurrent requests. App Runner scales toward idle when quiet (cheap); Fargate autoscaling adds tasks under load.
  • Database — the first thing to feel load. Lever = a bigger RDS instance (micro → small → medium). One knob.
  • LLM — scales linearly with questions asked. At any real usage this is the dominant cost, dwarfing infra — cap it with per-user/day token limits and the cheap-model fallback (already in the design).

💸 The "cheap first, improve later" reading

For a v1 that just needs to feel like the real thing, App Runner ≈ $30–50/mo is the floor — no ALB, no NAT, scales down when idle. You can graduate to the hardened Fargate topology (private agent, WAF, custom domain) later without touching app code — it's the same images and the same Terraform, just different service definitions.

The honest caveat: whichever you pick, turning on the real LLM agent will out-cost the whole infra bill fast. Infra is the cheap part.

✔ You're fine with cold starts — so we run App Runner at its minimum 1-instance, smallest-size tier (see the warm/cold split above). Cheapest config, perfect for a test/experience build.

👥
How many concurrent users? App Runner handles ~100 concurrent requests per instance and autoscales to 25 instances by default (raisable on request) → low thousands of concurrent requests. ECS Fargate scales by task count behind the ALB → effectively unbounded. But for this app, neither is the real ceiling — the practical limit is the LLM (each question holds a request open for seconds while the model thinks) and the RDS connection pool. Those cap useful concurrency long before the container platform does, so App Runner's headroom is far more than a v1 needs.

7 · The build plan

Six phases. Auth (B) is deliberately first and done locally so it's de-risked before any cloud spend. Each phase ends with something you can verify.

  • A · AWS foundations (Terraform skeleton)

    Bootstrap: Terraform state (S3 + DynamoDB lock), VPC (public/private subnets, NAT), ECR repos, RDS Postgres 16 (enable pgvector), Secrets Manager entries, and the GitHub-OIDC IAM role for CI. No app yet.

    ✔ Verify: terraform apply succeeds; you can connect to RDS and CREATE EXTENSION vector;.

  • B · Google auth rework local-first

    Implement auth_mode=google end-to-end (§4): backend GoogleVerifier + JIT, /auth/config, frontend Google button, the users migration, and the Google OAuth setup — creating the OAuth 2.0 client, configuring the consent screen, and registering redirect URIs (yes, that's in scope here, per the §4 change table). Keep the dev stub as default.

    ✔ Verify: locally flip AUTH_MODE=google, sign in with a real Google account, /me returns the JIT-provisioned user, RLS still isolates.

  • C · Build & ship images + static frontend

    Production frontend build → S3 + CloudFront (replaces the dev-server container). Confirm the 4 Dockerfiles build for linux/amd64; push backend-api / data-agent / db-migrate / pipeline to ECR. Upload source CSVs to the S3 data bucket; point the pipeline at S3 instead of the bind mount.

    ✔ Verify: images in ECR; the static frontend loads from CloudFront.

  • D · Deploy services + run jobs

    Terraform the two App Runner services (backend-api public, data-agent as its downstream), secrets injected from Secrets Manager, DB_SSL=require, CORS + Google redirect set to the CloudFront URL. Run the migrate then pipeline one-off ECS tasks against RDS — loading the full datasets from S3.

    ✔ Verify: hit the API health + /auth/config in the cloud; full marts populated in RDS.

  • E · CI/CD — one workflow, push-button

    New deploy-aws.yml: OIDC into the IAM role → build/push all images to ECR → terraform apply → run the migrate task → force ECS new deployment → run the smoke test against the live URL. Keep ci.yml as the PR gate (lint/test/build/journey-evals).

    ✔ Verify: a merge to main deploys with no manual steps and the cloud smoke test is green.

  • F · Domain, TLS & harden

    Route 53 hosted zone + ACM cert → custom domain on CloudFront & ALB (HTTPS). Add WAF, statement timeouts, per-user LLM cost caps, CloudWatch alarms/dashboards. Finalise Google redirect URIs for the real domain.

    ◻ The "use it exactly like local" milestone: open the domain, Google login, ask a question, admin dashboard — all working.

9 · Risks & open questions

All four decisions are now locked (App Runner · Google-direct · CloudFront URL · full data). The residual risks below are things to handle during the build.

Q1 ✓ decided Compute

App Runner (~$30–50/mo, §6) for the cheap v1 — harden to ECS Fargate later without touching app code.

Q2 ✓ decided Auth

Google direct (§4) — reuses the existing OIDC verifier + JIT pattern, no new infra. Cognito stays available later if you ever want multiple IdPs.

Q3 ✓ decided Domain

Default CloudFront URL for v1 — cheapest, ship the experience now. A custom domain + TLS (Phase F) is deferred until you want it; Google redirect URIs will use the CloudFront URL to start.

Q4 ✓ decided Data

Full ~580MB datasets in the cloud. Implication: the CSVs go to S3, the pipeline run-task gets more memory + a longer timeout (~8 min, ~3M rows), and RDS storage is bumped to ~20 GB gp3 (a couple $/mo). Baked into Phase C/D.
