Streaming Pages

Make the data-agent stream its answer page by page over the existing SSE channel — Page 1 (Summary) lands and renders while the agent is still working on Pages 2+ — and give admins the exact render JSON (the Template Studio contract) inside the chat “agent run” expander. The wire shape of a page is byte-for-byte the same Page JSON Template Studio already teaches.

Where we are today (verified in the code)

Streaming exists — but only for steps

Chat already calls POST /ask/stream (api.ts:373) → backend-api relays the data-agent’s SSE (routers/ask.py:313). Frames today: status heartbeats, progress ({n, action, detail} from deps.emit()), then one result with the entire answer. The user watches a step list until everything lands at once.

Pages are composed at the very end

compose_pages() (agent/pages.py:231) derives the Summary + Insights pages from the finished InsightReport — called after agent.run() completes (sandbox_agent.py:301) and on the stub path (main.py:423). The model itself builds the whole report in one run_analysis pass.

The contract JSON is admin-visible only in Template Studio

ContractJson (features/admin/ContractJson.tsx) shows “what Data-Agent sends the frontend to render this page” — elided rows + copy-full — but only under the Template Studio tab. The chat “agent run” expander (AgentTrace) shows steps and decisions, never the render JSON.

One shared renderer already guarantees parity

PagesView and Template Studio both render through PageLayout + the template registry, so any page we stream mid-run can reuse the exact same component — streamed pages can’t drift from the final answer or from Studio previews.

The gap is composition timing, not plumbing. The SSE pipe, the queue (_SbDeps.progress), the relay, and the page renderer all exist. What’s missing: (1) per-page composers that can run before the whole report exists, (2) two new frame types on the wire, (3) a page plan so the frontend knows how many pages are coming, and (4) the render-JSON surface in chat for admins.

Target flow — plan first, then pages in order, result last

One question, four kinds of frames. The plan frame declares up front how many pages the answer will complete; each page frame carries a fully validated Template-Studio Page; result stays the single authoritative, persisted answer.

The answer strategy — one pull of data, layered pages

Every answer follows the same shape: the agent pulls all the data it needs once, then each page is a deeper analysis pass over those same frames. No new SQL between pages.

Extract once, wide. The extract pulls the metric at the grain the question needs plus the explanatory attributes the mart offers (e.g. bedrooms, property_type for rents) — aggregated in SQL to month × entity × attribute so it stays well under the row cap. deps.frames already persists across run_analysis calls, so pass 2 slices the same DataFrame for free.

Page 1 · Build trust

Aggregate the frame to headline level: set the values — latest value, growth, the trend chart. Answers the question directly and fast; this is what the agent provides today.

Page 2 · Explain the values

Slice the same data by the extra factors — driver_analysis / comparison_chart over bedrooms, property type, price band… — naming what moves the Page-1 numbers, when it’s relevant to the question. No relevant attributes → page 2 is skipped explicitly.

Page 3 · Act on it M4

Opportunities derived from the same frames — where the growth/yield story says to look next. Same rule: no new extract, deeper analysis.

Page plan policy — who decides how many pages

The plan is deterministic policy in the data-agent, not a model choice — the model can’t promise pages it might not deliver, and the count must be known before the run starts so the frontend can draw the ghost slots. Gating is per user (decided in D2): a plan column on app.users (free | plus | pro, migration 0016), seeded admin → pro, user1 → free, user2 → plus. backend-api loads it with the current user and passes it through AskRequest.user.plan; the data-agent’s page_plan(plan) does the rest. Pages above the user’s plan appear in the plan frame as status:"locked" teaser entries (decided in D3).

User planPlanned pagesLocked teasersNotes
free (user1) 1 · Summary 🔒 2 · Insights, 🔒 3 · Opportunities The answer-at-a-glance page — what the agent provides today, condensed. The model runs pass 1 only (no insights pass), so free questions also cost fewer turns.
plus (user2, paid) 1 · Summary → 2 · Insights 🔒 3 · Opportunities Today’s full behaviour, streamed.
pro (admin, max) 1 → 2 (→ 3 when M4 ships) none Page 3 needs real analysis skills (yield gaps, momentum ranking) — M4. Its template reuses insights/two-col until a dedicated one earns its place (adding one touches the 3-way registry sync: registry.ts + pages.py + migration seed).
A planned page that produces nothing is skipped explicitly (page {status:"skipped"}) rather than left hanging — the frontend removes the skeleton. result.pages only ever contains completed pages, exactly as today.

Two-pass analysis — Page 1 genuinely early feature ~1 day

Failure modes & compatibility

ScenarioBehaviour
LLM fails after plan/page 1 streamed Salvage/stub path emits its own plan + pages (replacing by index); result is authoritative — the frontend always reconciles streamed slots against result.pages, so a divergent stream can never leave a stale page on screen.
no_answer run Plan collapses to a single summary slot; page 1 is the honest-reason text object (or classic report view on reconcile). No skeletons left behind.
Model ignores the two-pass instruction (M2) Pass 1 that builds a full report just triggers both page emits back-to-back (M1 behaviour). No pass 2 → page 2 skipped; whatever insights exist at result time still persist in the report.
Non-streaming clients (POST /ask, API channel, smoke fallback) Untouched — plan/page frames exist only on the stream endpoints; AgentAnswer/AskResponse schema is unchanged.
Persistence / history reopen Unchanged: pages stored in messages.report.pages, trace in query_runs. Reopen renders final pages exactly as today (no replay of streaming).
Feedback / evals Element ids and role semantics unchanged; feedback is only enabled once result supplies message_id, same as now.
backend-api read timeout (30s) Page frames are extra bytes on the wire between the existing 2s heartbeats — strictly safer, no timeout change needed.
Old frontend against new agent Unknown SSE events are ignored by the existing parser loop — forward-compatible by construction.
Wide extract would blow the row cap (too many attribute combinations) The prompt tells the model to aggregate in SQL to month × entity × attribute (small by construction) and to drop attribute dims rather than truncate the series; a metric-only extract means pass 2 has nothing to slice → page 2 skipped honestly.
User with no plan value (legacy rows, API channel) plan defaults to free in the column; backend always sends a value, and the data-agent falls back to free if it’s missing — the safest (cheapest, least-revealing) behaviour.
