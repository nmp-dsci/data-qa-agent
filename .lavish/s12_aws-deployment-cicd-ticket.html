4 · Cost/ops levers

Database is now decided (Aurora Serverless v2); tracing stays optional.

🗄️ Aurora Serverless v2 ✅ chosen

Aurora Serverless v2 scales to zero / auto-pauses — ~$0 compute when idle, and data persists through the pause. pgvector supported.

  • Why for this app: a personal test project that's idle most of the time → pay ~$0 while it sits, not a flat ~$15/mo.
  • Trade-off accepted: ~10–15s resume latency stacks on the App Runner cold start → a slow first hit after quiet spells (fine for a demo).
  • Not run locally — local stays on the Postgres container; both rebuild from the same Alembic migrations, so it's fully reproducible / destroyable (terraform destroy).

Set min ACU low (e.g. 0 auto-pause / 0.5 floor) so idle cost stays near zero; bump the floor later if first-hit latency bites.

📊 Tracing without Logfire Cloud

You already trace into the DB: every /ask writes app.query_runs (trace + SQL + tokens/latency) and app.events — that's what the admin dashboard reads.

  • Logfire is optional and OpenTelemetry-based → export can point at CloudWatch / X-Ray (serverless) or dump spans to S3 (cheap, query with Athena).
  • Default for v1: keep the DB audit trail (already built, $0) + CloudWatch for container logs; skip Logfire Cloud.

⌨️ Setting the one secret that's yours — from the terminal, not the console

The LLM API key is the only value you place by hand (Phase D). No console click-ops needed — Terraform creates the empty secret container first, then you set the value with one command:

aws secretsmanager put-secret-value \
  --secret-id data-qa/llm-api-key \
  --secret-string 'sk-...your-key...' \
  --profile data-qa --region ap-southeast-2

The key never touches git or chat — it goes straight from your terminal into the vault. Then I confirm it works: I redeploy the service and hit its health/ask endpoint to prove it read the key. DB password + JWT_SECRET need none of this — Terraform generates those into Secrets Manager itself.
