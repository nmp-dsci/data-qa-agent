TL;DR

The engine is good. The frame is what feels unprofessional.

What's strong: the report engine (KPI tiles + visx charts + page templates), the streaming ghost-page UX, the two-layer design-token architecture, ⌘K palette, and a11y basics (focus rings, ARIA tabs). Template Studio proves the product's output already renders beautifully.

What breaks the impression: three structural layout bugs make the app feel broken before styling even matters — the page scrolls as one giant document so the composer and header scroll out of view, nothing auto-follows the streaming answer, and on a phone the fixed 230px sidebar leaves ~140px for the actual chat. On top of that: a repo slug (data-qa-agent) as the brand, system-default typography, browser-style tabs, answers trapped in bubbles spanning ~1200px, a bare login card, and no motion.

The plan: keep the stack and the report engine; rebuild the frame in 6 shippable phases — A layout/scroll fixes · B identity + tokens v2 · C app shell (icon rail, mobile bottom nav) · D chat canvas (hero composer, open answers, reading column) · E login & first-run · F parity + responsive hardening with a Playwright viewport-regression suite.

MOVE 1

Contain the layout

Height-capped shell; panes scroll, composer and nav never leave the screen; stream auto-follows.

MOVE 2

Reading column

MOVE 3

MOVE 5

True responsive

3 tiers (desktop / tablet / mobile): icon rail → bottom nav, drawers, dvh + safe-area, 44px targets.

SECTION 2

What the best apps do — and what to take from each

Researched against current (2026) production patterns in ChatGPT, Claude.ai, Perplexity, Linear, and the modern data tools (Hex, Omni). The target in one line: Perplexity's ask flow, in a Linear-quality shell, with ChatGPT's reading ergonomics — wrapped around the report engine you already built.

ChatGPT · Claude.ai
Shell & reading ergonomics
  • One collapsible left rail (~260px) holding nav + date-grouped history; collapses to icons, becomes a drawer on mobile.
  • Message column capped ≈768px — long answers stay readable at any window size.
  • Assistant answers on the canvas, not in bubbles; only the user's message is a bubble.
  • Composer: auto-growing textarea, Enter sends / Shift+Enter newline, stop button while streaming, sticky at bottom with a fade.
  • Auto-follow during streaming with a "jump to latest" pill when you scroll up.
Perplexity
The ask flow
  • Hero composer: on empty state the input sits centered mid-screen with suggestion chips — the page IS the invitation to ask.
  • Docks after first ask — composer moves to the bottom and the answer takes the stage.
  • Answer as a document: title, meta row (sources → for us: engine, rows, SQL), then content blocks.
  • Follow-up chips under every answer keep the conversation going.
Linear
Craft & consistency
  • Token rigor: one type scale, 4/8px spacing grid, hairline borders, 3 elevation levels — nothing ad-hoc.
  • Motion: 120–200ms ease-out on everything interactive; nothing cuts, nothing bounces.
  • ⌘K as spine (you have this!) plus visible keyboard hints to teach it.
  • Skeletons everywhere — content-shaped loading, never spinner text.
Hex · Omni
Data-answer presentation
  • Results are artifacts: each output is a card with a toolbar — copy, CSV, PNG, open-in-editor.
  • Numbers set in tabular figures, right-aligned in tables, units in the label not the value.
  • Chart cards carry title + basis/source line + export — exactly your KPI/chart card shape, formalized.
  • Schema tree with search and inline column profiles (you have the tree; keep it).

SECTION 5

Responsive strategy — three tiers, one shell

Replace the single 720px patch-query with three deliberate tiers. The same shell component renders all three; only the nav chrome and panel behavior change.

≥ 1024px

Desktop

  • 56px icon rail (brand / nav / theme / user)
  • Contextual panel: conversations (chat) or catalog (SQL), collapsible, 260px
  • Thread reading column ≈768px; report cards may breakout to 960px
  • Content max 1440px centered (as today)
768 – 1023px

Tablet

  • Rail stays (56px); contextual panel becomes an overlay drawer
  • Reading column fills width minus rail, max 768px
  • Report pages keep 2-col templates; KPI grid 2-up
  • SQL: catalog as drawer, editor full-width
< 768px

Mobile

  • Bottom tab bar (56px + safe-area); rail hidden
  • Conversations = slide-over sheet (☰)
  • Composer docks above nav; 16px input font
  • Report columns stack; charts capped 240px; tables scroll-x with sticky first column

SECTION 7

Risks & migration concerns

e2e selector breakagehelpers.ts logs in via getByRole("tab") and text clicks; Phase C changes nav semantics. Mitigation: update helpers in the same PR; smoke + chat/studio specs are the contract.
Shared renderer blast radiusPageLayout is used by chat, Template Studio and the playground — restyles propagate everywhere. Mitigation: that's by design; add studio + playground to the visual suite in Phase F and eyeball in Phase D.
Template registry couplingColumn tracks come inline from the registry with an !important mobile override. Keep that contract in Phase A/C; revisit only if a template needs a tablet-specific layout.
Streaming abort pathThe stop button needs askStream to accept an AbortSignal; verify backend closes the SSE cleanly and the conversation stays consistent (message persisted or not at all).
Font loadingSelf-host via @fontsource (no CDN, works in the App Runner deploy); preload + size-adjust fallback metrics to avoid CLS; ~60KB gz added — acceptable.
Light theme driftEvery new token needs a light value or light mode regresses quietly. The visual matrix runs both themes to catch it.
Conversation rename/delete needs backendTwo small endpoints (PATCH title, DELETE) + RLS checks. If out of scope, sidebar v2 ships with grouping/search only — still a big win.
Known live-stack flakeDuring this review the live agent didn't return a report within 4 min (matches the undiagnosed journey failures noted in CI memory). Unrelated to UI but will make Phase D verification annoying — worth a look first.
