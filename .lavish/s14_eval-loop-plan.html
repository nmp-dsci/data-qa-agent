The Eval Loop

A plan to prove the data-agent can be trained by evaluation, in three phases: ① build the golden answer for a question (SQL · sandbox · pages — agent-drafted, then human-confirmed); ② run evaluations that batch-score the agent against those goldens across every stage; ③ a teacher agent reviews the scored evals vs the goldens and proposes data-agent updates to improve — then re-scores. Golden-first, so there's always a 100/100 benchmark to train toward.

01 · What "train the agent" means here


GOAL A · EXTRACT

SQL — pull the right data

The governed SQL extract selects the right rows: correct scope, timeframe and grain, under the asking user's RLS — regardless of which query path got there.

Proved by G1 · extraction grader — golden SQL executed under the same RLS, values compared with tolerance.
GOAL B · PREPARE

Sandbox — build the metrics (the plan)

In the sandbox the agent turns the extract into the answer: the right derived metrics and data-prep (yield, YoY, ranking) via tested skills — what it computed and how.

Proved by G2 · preparation grader — the sandbox plan + derived metrics vs the golden prep.
GOAL C · PRESENT

Frontend — communicate it back

The delivered pages conform to the UI contract and explain the answer: right objects, grounded numbers, the why + a so-what — not a restated chart.

Proved by G3 · presentation grader — PagesEnvelope format + the insight judge (rubric /10).

"Train" ≠ fine-tune (in v1). The agent already has versioned, editable behaviour levers: the Insight Playbook knowledge tree (knowledge_version content-hash is stamped on every report), the skills library, the system prompt, and the model config. An eval loop trains the system by making one targeted lever change per cycle and proving the score moved. Crucially it doesn't just score — a failure clustered to a stage points at the lever: extraction wrong → schema/knowledge guidance; weak prep → a skill or Playbook method page; weak presentation → a system-prompt or template edit. The same loop is the acceptance gate for actual fine-tuning or automated self-improvement later — build the rails first.

04 · E1 — the golden pack


One YAML pack per dataset (evals/cases/nsw_property.yaml), matching the dataset-agnostic direction from s05 — a future dataset ships with its own eval pack the same way it ships its own marts. Cases are grouped in 7 tiers (dataset-agnostic), each proving a different capability — you author as many per tier as you need (no fixed target; D1 scrapped).

Tier#What it provesExampleGraded by

T2 · trend & grouped12 Time grain + group-by handling; the trend object reaches the UI with the right series "Rent by month for Hornsby by property type and bedrooms" G1 series match · G2 object:trend · G3

Case format — extends the expectation vocabulary the two existing YAML suites already use, plus a golden block for ground truth. Authored in the Evaluations tab (§07); this YAML is the committed export of what the DB holds:

# evals/cases/nsw_property.yaml — one case
- id: yield-top-suburbs
  tier: 3
  tags: [yield, join, ranking]
  as_user: user1
  question: Which suburbs have the best rental yield?
  golden:                     # three executable stages — replayed at grade time
    sql: |                    # ① EXTRACT — hand-verified once, re-run under the case's RLS
      SELECT suburb, ...
    sandbox: |               # ② PREPARE — run_analysis script over the extract
      yield = annual_rent(df) / median_price(df)
      out   = rank_desc(yield, k=5)
    kind: ranked_set  key: suburb  k: 5  tolerance_pct: 1.0  # how values compare
    report: golden_report.json  # ③ PRESENT — PagesEnvelope; objects bound to the prepped data
  expect_format:            # reuses report_evals vocabulary + page-level checks
    [report_well_formed, pages_valid, has_headline, has_insight]
  judge_focus: should explain what yield means and why these
    suburbs rank (rent level vs price level), not just list them
  holdout: false            # ~20% marked true: scored, never used to steer fixes
Can the YAML regenerate the answer, and how is it enforced? Yes — the golden is executable end-to-end and replayed at grade time: ① the sql runs under the case's RLS → ② the sandbox script runs on that extract in the same locked-down sandbox and must reproduce the metrics → ③ the report validates against PagesEnvelope and every object's data must resolve to the sandbox output. Enforcement is a pack-lint CI test: it EXPLAINs the SQL, dry-runs the sandbox script, and validates the pages bind to the produced data — so a golden that can't regenerate its own answer fails CI before it is ever used to grade the agent.

Where cases come from

  • Author from scratch in the Evaluations tab (§07) — the primary path when a dataset is new.
  • Migrate the 8 existing journey/report questions in as T1–T3 seeds.
  • Mine query_runs for real questions asked in dev — the traces show which ones the agent struggled with.
  • Generate variations with Opus (paraphrases, harder scopes), then hand-verify each golden SQL once.
  • Promoted feedback: active app.eval_cases rows join every run as judge-graded cases — their free-text expectation becomes the judge instruction. The runner is what finally increments their staleness lifecycle.

Ground truth runs under the same RLS

The marts are row-level-secured, so golden SQL must execute in the same visibility the agent had: the runner sets app.current_user_id to the case's as_user before running the golden query. Running goldens as a bypass role would compare against rows the agent legitimately can't see — T5's isolation cases depend on this.

Values are recomputed at grade time against the committed deterministic sample data (data/samples/), so goldens don't rot when marts are rebuilt — and a pack-lint CI test EXPLAINs every golden to catch schema drift early.

05 · E2 — the four graders


GraderTypeWhat it checksPass rule
G1 · Extraction — SQL deterministic Did it pull the right data? Executes the case's golden SQL (same RLS context), compares against the agent's extracted rows / report values. scalar: tolerance · row_set: F1 on key columns · ranked_set: top-k overlap · series: per-point tolerance. Grades the numbers, not the SQL text — any valid query path is fine. score ≥ 0.8, exact for T5 denials
G2 · Preparation — sandbox plan det + light judge Did the sandbox build the answer correctly? Compares the agent's derived metrics / prepped frames (yield, YoY, ranking) against the golden prep, and checks the plan — which skills it called and how — is sound. Reads the run_analysis steps straight from the trace. metrics within tolerance · sound plan
G3 · Presentation — frontend deterministic + judge Does it conform and explain? Deterministic: report_structural_issues() == [] · PagesEnvelope validates · expected object types present · element_id/explains wiring resolves. Judge: the insight rubric below, scored /10 by a different model family (frozen prompt, evals/judge/insight_rubric.md) with a numeral-groundedness pre-pass. zero format issues · insight ≥ 7/10
G4 · Ops recorded Model turns, input/output/cache-read tokens (input is cache-dominated — billed cost tracks turns, so turns is the headline number), latency, guardrail events. Recorded per case from the trace; not gated in v1, budget gates later. — (trend only)

After G1–G4: the teacher agent. Grading runs in batch — G1–G4 score every case — then the whole batch (scored results + agent outputs + goldens) is handed to a teacher agent (Claude Agent SDK, strongest model) that reviews the scores against the goldens and proposes changes to the data-agent to raise accuracy (§09). Graders produce the numbers; the teacher turns them into an action, then the pack is rescored.

G3 rubric — 5 × 0–2 points

0–2Grounded — every number traceable to query rows (pre-pass assists); one fabricated number caps the whole score 0–2Direct — answers the asked scope/timeframe, states assumptions when the question was vague 0–2Explains why — drivers, decomposition, or comparison vs a baseline; not the chart restated in prose 0–2So-what — an implication or recommended next look 0–2Clear — plain language, no filler, headline actually headlines

Judge discipline

  • Calibrate before trusting: hand-score ~10 real answers first; require directional agreement before the judge gates anything.
  • Version everything: judge prompt hash + judge model recorded on every result row, same as knowledge_version.
  • Cross-family: agent runs DeepSeek (default env) → judge runs Claude, avoiding self-preference.
  • Batch, not 1-by-1: the judge runs as a single Claude Agent SDK pass over the whole set (all agent-output vs golden pairs at once), not per-case LLM calls — cheaper, and it can reason across cases.

Persistence & the agent-version stamp — migrations 0019–0020

Every run — chat or eval — must record which agent produced it. Today knowledge_version() (a content hash of the Insight Playbook, agent/knowledge.py) is the only behaviour hash on a report, and the provider/model live only in agent/config.py. That is not enough to trust a base-vs-proposed comparison, because the provider, model, system prompt and skills library can all change too. So 0019 introduces a first-class agent_version descriptor and stamps it on query_runs; 0020 adds the score store that references it. Eval asks still tag channel='eval', so every case's full trace lands in query_runs and eval_results.query_run_id joins straight to it — failure diagnosis stays inspect_run.py <id>.

06 · E3 — prove the loop (the actual milestone)


The deliverable you asked for is not "an eval suite" — it's evidence the loop trains the agent. That evidence is two documented improvement cycles:

  1. Baseline. Full pack against current main → eval_run #1 at knowledge-version kv-A. Triage the 4 known live-stack journey failures into the clusters while at it.
  2. Diagnose. Cluster failures by tier × grader × tag; read the traces (inspect_run.py) for the biggest cluster. Hypothetical example: "T3 yield cases fail G1 — the agent averages monthly yields instead of annualising rent over median price."
  3. Intervene — one lever only. e.g. author a knowledge page knowledge/rental-yield-method.md (or fix a skill). The Playbook hash changes → kv-B. Log hypothesis + files in evals/interventions.md.
  4. Re-run. Same pack, same judge → eval_run #2 at kv-B.
  5. Compare + gate. compare.py: target cluster must improve, and no case flips pass→fail anywhere else. Auto-generate docs/evals/cycle-001.md from the two runs.
  6. Repeat once on the next cluster → cycle-002.md. Two green cycles = the loop is proven; from then on it's a flywheel that user feedback keeps feeding.

Tab 1 · Golden Answer (Builder) — the creation flow

You don't start from a blank page. The tab proposes questions for the dataset (with variations); you pick one and hit Run, and the data-agent pre-populates all three golden stages as a first draft — SQL, sandbox plan, and pages. You then confirm the golden by correcting the agent's answer stage by stage, re-running just the segment you changed (a change cascades to the stages below). The result is "what a great answer should render as," built with the same three surfaces the agent uses:

MILESTONE 1

Ask & confirm the SQL

Ask the question as a chosen user (sets the RLS context), then inspect the agent's SQL or write your own, run it, eyeball the rows, and lock it as the golden query.

Reuses /ask + the SQL editor (features/sql/SqlEditor.tsx — CodeMirror, runSql, catalog). → golden_sql + as_user, verified under RLS.
MILESTONE 2

Sandbox preps the data

The confirmed SQL becomes the extract; the same sandbox that runs the agent computes the derived metrics/data-prep (yield, YoY, ranking) — so the golden's numbers come from tested skills, not hand-typing. The admin can edit the prep code and re-run until it's right.

Reuses the locked-down sandbox (agent/sandbox_agent.py — "extract → run_analysis → report"). → golden_data (frames + metrics), deterministic.
MILESTONE 3

Compose the pages

Hand-build the report — summary page, insights page, more — placing KPI/trend/breakdown objects into template columns, exactly as the agent would. Live preview renders through the production layout.

Reuses Template Studio / Playground (features/admin/TemplatePlayground.tsx → same PageLayout as chat; PagesEnvelope contract). → golden_report = the ideal answer.

09 · E6 — auto-research, later (what these rails enable)


deliberately out of scope for v1

Once the data-agent has been scored against the goldens, batch all the results to the teacher agent — a Claude Agent SDK on the strongest available model (Opus-class), not a single LLM call. Given read access to every eval_run + eval_results + the joined query_runs traces, it reviews the diagnostics across the whole batch (every agent output vs its golden) → proposes updates to the data-agent (one lever: a knowledge page, skill, or prompt) → a candidate run rescores → the same regression gate decides → an accepted change arrives as a PR with the cycle report. A human reviews the diff; the evals are the specification.

Prerequisites this plan intentionally creates: a broad-enough pack (~80+ cases), a calibrated cross-family judge, the agent_version stamp so a proposal is provably one lever, per-run cost caps, and the one-change-per-cycle discipline already encoded in the tooling. A read-only diagnosis assistant — the same SDK agent with no write access — can ride these rails during E3 immediately, surfacing cluster hypotheses that a human still acts on.
