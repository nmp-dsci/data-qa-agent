s16 · Golden Examples builder

Editing one object should update SQL → sandbox → presentation together — not destroy the rest

In the Golden builder, “✎ edit object → describe → Generate & run” currently rewrites the whole golden_sandbox for a single object, silently deleting the pipeline for every other object in the golden. This plan makes the edit a coordinated, non-destructive cascade that keeps the four stored stages consistent.

✓ Scope chosen: SQL + sandbox + presentation (full cascade)

01 The break — reproduced

I authored a 2-object golden (a KPI “Peak median” + a Line + Bar chart) whose run_analysis builds both — skills.latest_value(…) + headlines=[…] for the KPI and skills.dual_axis_chart(…) for the chart — then edited only the chart via the AI box and pressed Generate & run.

Why it “breaks”: every /admin/eval-goldens/object call returns 200 OK and the chart re-renders — nothing throws. But the golden is now internally inconsistent: the presentation still shows a KPI that the sandbox can no longer produce. The next ▶ Run skills, reload, or eval-runner regeneration drops the KPI (and any other non-edited object) because its backing pipeline is gone.

Root cause

The per-object AI edit applies a single-object script over the whole golden. Two lines in the parent do the damage:

frontend/src/features/goldens/GoldensPage.tsx · instructObject() lines 478–479. res.code/res.report come from services/data-agent/agent/object_codegen.py · scaffold_object(), which by design writes a script that builds only this object’s main_chart — it has no knowledge of the golden’s other objects, and never touches the SQL extract.

02 Target behavior — the cascade

Describing a change to one object triggers a single governed pass that may touch all three upstream stages, then re-renders the object in place while keeping every stored stage reproducible.

Non-destructive by construction. The AI is handed the current run_analysis and asked to modify it minimally — so the other objects’ code (and data) survive. The presentation keeps the curator’s layout: only the data of matched objects is refreshed, and only the edited object is fully replaced.

03 Proposed approach

Engine: an agentic authoring loop, not a blind one-shot call

The current path is a single structured LLM call that writes code without running it — so a wrong column or bad skill call surfaces as an error the curator has to retry. We give the model tools and let it close the loop: verify the extract, test the run_analysis, self-correct, then finalize.

Rejected · one-shot

Single structured LLM call

Fast (~8s) & cheap, but blind: guesses column availability from a name list, can’t see sandbox errors, no self-correction. Weak fit for “understand what it needs to do”.

Chosen · agentic loop

pydantic-ai agent + tools

The existing pydantic_ai.Agent gets two tools — run_extract(sql) and run_sandbox(code) (thin wraps of the in-process extract()/run_code()). It inspects real columns, tests the chart, fixes errors, then returns verified {sql, code}. Bounded by a turn cap.

Why not the Claude Agent SDK? It’s Anthropic-only, and this env’s live provider is DeepSeek (choose_provider(settings.llm_provider, …)). The whole codegen stack already runs on pydantic-ai, which tool-calls across both providers — so we stay provider-agnostic and don’t fork the agent stack. Cost note: a loop adds turns (the turn-dominated dimension), acceptable for an occasional authoring action and capped.

The fix, at the point of damage

04 Risks & failure modes

🔒 Auto-editing the golden’s SQL

The SQL is a hand-curated stage. An AI rewrite could subtly change the extract (filters, joins) and shift all objects. Mitigation: the agent verifies the new SQL via run_extract before returning; it runs through governed extract() (RLS + safety); the old SQL is kept on failure and behind a one-click revert banner (Q1).

🧩 Objects the sandbox can’t represent

A hand-composed golden with multiple charts beyond main_chart + one insight chart can’t all be reproduced by a single run. Those keep their existing rows (no refresh); only matched ids update. Documented as a known limit, not a crash.

🆔 element_id matching

compose_pages emits stable ids (headline:i, report:chart, insight:i:chart). Drafted goldens match cleanly; curator-added edit:* objects don’t and are skipped by design.

↩️ No partial writes

Any failure (bad SQL, sandbox error, no chart) returns error with the stored stages unchanged — the current clobber is replaced by an all-or-nothing apply.
