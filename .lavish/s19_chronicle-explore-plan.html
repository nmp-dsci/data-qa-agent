02 · CONTEXT

How the legacy tool works — and what actually needs porting

The pre-computed cube (why it exists)

GitHub Pages has no backend, so profile_nswgov.py / profile_rentboard.py pre-aggregate every combination of up to 4 dimensions into one long table, keyed by cutid, serialized as JS globals:

data_df = [{ cutid:"7", agg0_c:"bedrooms", agg0_v:"3", agg1_c:"postcode", agg1_v:"2287", agg2_c:"rent_yyyymm", agg2_v:"2021-06", agg3_c:"all", agg3_v:"all", rental_n_v: 41, rent_n_v: 18130 }, …] lookup_df = [{ cutid:"7", value:"all___bedrooms___postcode___rent_yyyymm" }, …]

The browser resolves the user's filter selection → sorted dim signature → cutid → slice of rows, then pivots/derives client-side (e.g. rent_a_v = rent_n_v / rental_n_v). In Data Pilot none of this is needed: Postgres does the aggregation live.

Legacy scope was deliberately tiny to keep JSON small — rent: Newcastle SA3 only; sales: North Sydney & Hornsby SA4 only, 2012–2021, RESIDENCE, ≤$8M. The app's marts cover all NSW.

The three tools (what users actually get)

TabBehaviour
ProfilingPick a Target cohort vs a Comparison cohort (e.g. FY2022 vs FY2021). Get: topline summary, metric deltas, per-predictor segment uplifts ranked (which bedrooms / postcode / region moved most), per-predictor charts, and a postcode choropleth.
TrendsTwo side-by-side chart builders: metric × month × split-dim → line or stacked bar, with dropdown filters. Instant pivot.
Data ExtractData dictionary (dims, classes, unique values) + pick up to 4 dims → aggregated extract → CSV download.

The real IP to port is the profile engine (profileCalc.js/profileMetrics.js: cohort deltas + segment uplift ranking) and the UX shape — not the D3 rendering (s07 locked visx-only) and not the cube.

  • No choropleth — visx kit has no map component (D2 locked: build in v1).

05 · ARCHITECTURE

One aggregation service, three consumers

The core idea: a single validated explore service in backend-api translates manifest-checked requests into parameterized SQL over the marts under RLS. The Explore UI, CSV extract, and (Phase E) the data-agent's tools all call the same service — so what users explore and what the agent can leverage never drift apart.

No cube. The legacy cutid cube was a workaround for having no backend. Marts are additive at the right grain; Postgres aggregates live in ms at this data size, with indexes already in place.
No client SQL. Unlike the SQL editor, Explore requests are structured JSON validated against the manifest — dimension and metric names are allow-listed, never interpolated from free text.
Agent-ready by construction. The profile engine ports profileCalc.js server-side once; UI and agent both consume it. Explore answers become reproducible objects goldens can reference (s18 lineage).
One chart library (locked in review). Every surface — Chat report pages, SQL editor previews, Golden Examples / Template Studio, and Explore — renders through the same shared components in frontend/src/ui/charts/ (Trend, Bars, Combo, KPITile), wired via report-engine/PageLayout.tsx. Explore ships no private chart code: line/bar reuse the existing components, the stacked-bar variant extends Bars, the choropleth lands as ui/charts/Choropleth.tsx, the comparison/ranked-delta tables land as ui/charts/DataTable.tsx, and new templates register as report-engine object types (table, choropleth) — so everything added for Explore is immediately usable by Chat reports, Goldens, and the agent, and the template set keeps growing in one place. The choropleth renders only for datasets whose manifest declares a geo binding.

The explicit rule: Explore's outputs are composed of report-engine objects — the same object contract the Golden Examples / Template Studio tab uses (today: kpi · trend · breakdown · compare · insight · text). Before building any visual, check the registry: if a template exists, use it; if not, add it as a new registered object type — never a one-off component. The s08 three-way registry sync test extends to cover the new table and choropleth types, so registry, renderer and templates can't drift.

07 · BUILD PLAN

Phases A–E — each independently shippable

PHASE A

Pipeline: geo dimension, yield mart + explore manifest

size M
  • New dbt seed postcode_geo.csv — postcode → SA2/SA3/SA4/GCC/state names, sourced from legacy tool_utils/util_postcode.json (ABS 2016 correspondences; ~2.6k rows).
  • New model marts.dim_postcode_geo — tagged agent_queryable; RLS: readable under either dataset grant (shared dim — needs a small extension of apply_dataset_rls or a permissive multi-dataset policy).
  • Explore manifest: per-dataset registry (dims, metrics incl. derived avg_* = total/n, labels, formats, geo-rollup joins, default views) as typed config in backend (app/explore/manifest.py), validated against the live schema at startup — same philosophy as dbt-tests-ground-the-agent.
  • Optional geo binding per dataset — {column: postcode, layer: poa_nsw}. The map renders only when the manifest declares a mappable column; a dataset with no geo column simply gets no map section. Keeps Explore dataset-agnostic (s05).
  • New marts.property_yield (pulled into scope by review) — sales ⋈ rent at postcode × property_type × year, from the property_yield_20241003.py logic: additive components (total_sale_value / n_sold / total_weekly_rent / n_rented) with gross_yield_pct defined as a derived ratio-of-averages — 52 × (total_weekly_rent/n_rented) ÷ (total_sale_value/n_sold) — so it re-aggregates correctly at any rollup. Registered as a third explorable dataset nsw_yield with its own apply_dataset_rls('nsw_yield') tag and grants, its own manifest entry (geo binding: postcode), and dbt tests: minimum cell volume (n_sold ≥ 5 and n_rented ≥ 5) plus a sane yield range.
  • dbt tests: dim_postcode_geo not-null/unique postcode; docs in _marts.yml.

PHASE B

Backend: /explore endpoints + profile engine

size M
  • GET /explore/datasets — granted datasets + manifest (dims, metrics, month range, dim domains capped at N values with typeahead fallback).
  • POST /explore/aggregate — {dataset, metrics[], group_by[], filters{}, months} → allow-listed identifiers only → parameterized SQL over the mart (+ geo joins) under rls_connection; row cap; audited like /sql/history.
  • POST /explore/profile — target + comparison cohort filters, response metric → port of profileCalc.js/profileMetrics.js: topline per cohort, metric deltas, per-predictor segment uplift ranking. Pure-python engine over two aggregate calls — unit-testable.
  • The engine also emits a per-predictor signal score (max |Δ| across its segments) — the UI orders the predictor charts strongest-signal-first, and within categorical predictors sorts segments by Δ (ordinal dims like bedrooms/month keep natural order, annotated).
  • POST /explore/ask — NL → tool state: LLM structured extraction constrained to the manifest (same provider abstraction as the agent, DeepSeek default; deterministic stub offline for tests). “compare FY2022 vs FY2021 weekly rent for houses” → populated Profile cohort rows / Trends selections. Output passes the same validator as hand-built requests; Profile is prefill-only (user hits Run), Trends may autorun (read-only aggregates).
  • Parity harness: scripts/explore_parity.py replays legacy cutid slices from docs/chronicle/datafeed/*.json against /explore/aggregate with the legacy region/date/purpose filters (needs make pipeline-full); additive metrics must match exactly.
