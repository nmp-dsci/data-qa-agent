One chart library, one object model — everywhere

Review of which surfaces render through the shared chart objects, why the Profile tables looked like a foreign format, and the plan to put Explore · Profile / Trends on the same page-object engine as Chat, Goldens and Template Studio — so any Profile output can become a golden answer or a chat page.

§1

The verdict — your statement, checked

TRUE
“Chat, Goldens and Template Studio use the chart objects.” All three render through the one renderer — report-engine/PageLayout.tsx header says it: “Chat answers (PagesView), the Template Studio preview, and the playground all render through this one component, so the layouts can never fork again.” The Golden Report editor (features/goldens/ReportEditor.tsx) imports it too.
HALF-TRUE
“Explore Profile / Trends are not on the same chart objects.” At the component level they are: Profile imports Bars, Choropleth, DataTable; Trends imports Trend, Bars — the same visx files Chat renders. But at the object-model level they are not: both tools hand-assemble bespoke JSX (ex-grid2 / ex-result-top / ex-card) instead of emitting Page objects through PageLayout. And Profile’s KPI tiles are raw ex-kpi divs — a second, parallel KPI format that isn’t KPITile at all.
FOUND IT
“Profile has tables and I can’t see the chart template table it used.” Profile’s tables are the shared component — ui/charts/DataTable.tsx (variants comparison / ranked, built in s19). You can’t see it as a template because table and choropleth were wired into the frontend renderer but never published as chart objects: they’re missing from the agent’s type contract (pages.py:54), from the Studio’s chart list (registry.ts CHART_OPTIONS), and from the goldens object builder. The component exists; the object is invisible.

Net: there is one chart library but two render paths, and the object registry is two types behind the renderer. Unifying = publishing table/choropleth as first-class objects, then making Explore emit pages instead of bespoke JSX.

§4

The six concrete gaps

#GapEvidenceConsequence
G1 Agent can’t emit table / choropleth agent/pages.py:54 — ObjectType = Literal["kpi","trend","breakdown","compare","insight","text"] vs lib/api.ts:114 which has 8 types A chat answer can never contain a ranked table or a map — even though the renderer supports both. Kills “chat leverages Profile structure”.
G2 Studio registry hides them registry.ts:103 — CHART_OPTIONS lists kpi/line/bar/grouped-bar/insight only Template Studio playground + Playwright matrix never show or exercise table / map / text. This is why you couldn’t find the table template.
G3 Goldens builder can’t author them agent/object_builder.py:350 — _BUILDERS = {compare, breakdown, trend, kpi} A curator can’t add a table or map to a golden via the named-object builder (s18 path).
G4 Profile KPI tiles are a fork ProfileTool.tsx:181–208 raw divs + styles.css:1126 .ex-kpi — not KPITile Two visual formats for the same object; KPI styling changes won’t reach Explore.
G5 Explore results aren’t pages ProfileTool.tsx:240–352, TrendsTool.tsx:141–213 — bespoke JSX layout, no Page/PageLayout A Profile run can’t be saved as a golden or reproduced by chat; layout logic forked from the engine.
G6 No object-type sync gate tests/test_registry_sync.py asserts template ids/columns only Object types already drifted once (G1 happened silently in s19). Nothing stops it happening again.

Bonus micro-gap found on the way: PageLayout.tsx:68–82 drops diverging when rendering a choropleth object — so a page-object map can’t render the Δ-centred colour ramp Profile’s selection map uses. Fixed in Phase A.

The plan — four phases

Ordered so the contract is complete and CI-gated before Explore is migrated onto it. Each phase lands independently green.

PHASE A

Publish table + choropleth as first-class chart objects

  • Agent contract: add "table", "choropleth" to ObjectType in pages.py + data-shape validation (table: columns[]/rows/variant/bar_key; choropleth: layer/key_field/value_field/rows/diverging) and prompt guidance for when the agent should choose them (ranked lists → table; per-postcode values → map).

§6

Risks & things that could bite

Interactivity loss on the Profile map

Choropleth in Profile may grow click-to-filter (onSelect) ambitions; page objects are data-only JSON. Mitigation: PageLayout already accepts a renderObject wrapper — Explore can wrap the map object with its own select handler without forking the renderer.

Agent quality, not plumbing

Giving the agent table/map types ≠ the agent using them well. Per the goldens-are-the-training-signal rule, we ship the capability + grounding and correct usage via goldens — no prompt-chasing.

Contract tests get stricter

The new object-parity test spans two languages via regex/AST like the template test — keep the patterns anchored (the s08 3-way sync test is the proven template).

Perf on the 616-shape map

ProfileResultView is memoized today so filter keystrokes don’t repaint the map. The pages subtree must keep an equivalent memo boundary (memo on the built Page[], keyed by result identity).
