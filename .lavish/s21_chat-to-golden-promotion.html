Step 2 · what happens on the click — a copy of stored artifacts, nothing executes
0 agent re-runs 0 SQL executed 0 LLM tokens

Nothing re-runs. The original ask already captured everything into the audit trail — the question and ① SQL in app.query_runs, the ② run_analysis script inside its stored trace, the ③ report pages in app.messages.report. The endpoint is a read-and-insert: it copies those stored artifacts into a draft golden. The exact answer you saw in Chat is what appears in Golden Examples — byte-for-byte, not a fresh answer.

Build plan — four small phases, one PR

P1

Backend · promote endpoint

~70 LOC · new endpoint
POST /admin/eval-goldens/from-run   (require_admin)
{ "run_id": "1565d54f-…" }
→ { "id": "<golden uuid>", "created": true|false }

1  SELECT question, sql_text, trace, message_id
     FROM app.query_runs WHERE id = :run_id        # 404 if missing
2  SELECT report FROM app.messages
     WHERE id = :message_id                        # pages = report.pages
3  code = _sandbox_code_from_steps(trace)          # existing helper, unchanged
4  SELECT id FROM app.eval_cases
     WHERE tags @> '["run:<run_id>"]'              # idempotency guard
     → hit ⇒ return { id, created: false }
5  INSERT app.eval_cases (question, dataset 'nsw_sales',
     tier 'T1', authoring_status 'draft',
     tags ['from-chat','run:<run_id>'],
     golden_sql, golden_sandbox, golden_data = report,
     golden_report = {'pages': pages}) RETURNING id

Accept · the endpoint is pure read-and-insert — no agent call, no LLM tokens, no SQL execution (steps 1–2 read the stored audit rows; the only write is the INSERT). POSTing run 1565d54f returns a golden whose copied ① SQL / ② sandbox / ③ report are exactly what the chat answer used, runnable as-is when the curator later presses Run; second POST returns the same id with created:false.
