Golden editing: make ② Presentation objects as easy as ① SQL

The Golden tab's three-stage editor works, but authoring a new presentation object means fighting a 15-field form with sales-only defaults, then hand-linking the result into the report. This plan makes a plain-English sentence the primary way to create an object — one instruction → sandbox object + report object, placed automatically. Most of the plumbing already exists; this is a UX-and-wiring change, not new machinery.

The worked example — your exact ask, Pre vs Post

"Average rent by month as x-axis, colour by bedroom band, filtered to house property type only."
Maps to marts.property_rent (month, bedroom_band, property_type; additive total_weekly_rent/n_rented) — and note skills.trend_series(..., group_col='bedroom_band') natively emits one coloured series per band, so the chart stack needs zero changes.

  2. Pick "Line chart"

  4. Fix dimension area_band → n/a for rent sales default

  9. Workaround: spot the collapsed "describe it in words" box, use the NL path instead undiscoverable
  10. Object lands in ② only → go to ③, open a column's ＋ picker, add a chart, bind via "Linked object" dropdown manual link
  11. Save

Build order

Phase 1 · the ask

NL-first authoring + auto-place

  • "New object with AI" panel leads ②; name auto-slugged, type keyword-guessed (both editable)
  • Auto-insert the lifted object into pendingPages + undo; never place when engine === "stub" or no object
  • name optional end-to-end (router → agent)
  • Demote the structured form under "advanced"

Small: one frontend panel + a place-into-pages helper + ~15 backend lines. No new endpoints, no migration.

Phase 2

Dataset-aware deterministic builder

  • Mart registry in object_builder.py (sales + rent; yield later)
  • defaultBuilder(dataset) — rent goldens open with valid defaults
  • Extend test_object_builder.py for the rent leg

Makes rent objects repeatable without an LLM — matters for eval reproducibility.

Phase 3

Polish + proof

  • e2e: sentence → chart in ② + ③ → Save (extend goldens.spec.ts)
  • Type inference refinement; placement target picker if Q1 wants it
  • Roadmap note: cross-golden object library (out of scope here)

Risks & failure modes

stub fallback

No LLM key / budget exhausted → scaffold_object returns the stub. Existing guard (GoldensPage.tsx:906) already refuses to apply it; the new auto-place must use the same gate so a stand-in object never lands in the report.

silent SQL rewrites

The NL build may extend ① to add missing columns. Keep the existing surface: "SQL changed (review / revert in ①)" + sqlRevert one-click restore. The extension is additive by prompt contract (keeps existing columns + WHERE), but the curator reviews it.

DeepSeek reliability

Already solved by design: this is the s16 orchestrated verify loop (≤3 bounded calls, we run the sandbox, feed errors back) — not an open tool loop, which DeepSeek thrashes in. No change to that architecture.

element_id churn

Renaming the auto-slug creates a new obj:<slug>. Keep upsert-by-element_id (existing upsertObject); rename before first Save is free, after Save it's a new object — same rule as today, worth a hint in the UI.
