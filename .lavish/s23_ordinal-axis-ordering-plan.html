3 · The decision: skills/data-knowledge — not golden+eval

OPTION A — data-knowledge + skills recommended

Encode the order once, centrally; the chart skills apply it; a deterministic control lets a curator override.

  • An ordinals registry (new, tiny): {area_band:[…], bedroom_band:[…]} with unknown last.
  • Chart skills auto-sort the x/category column by the registered order (+ an explicit x_order override).
  • Schema + knowledge carry the same order so the LLM also “knows” it when it writes SQL/sandbox code.
  • Generalises for free: every chart — chat, Explore, golden — orders bands right with zero per-answer work.
  • Goldens become the acceptance test that it renders correctly, not the mechanism.
OPTION B — golden-only + “learn via eval” rejected

Curator fixes each golden; the agent is supposed to “learn” the ordering through the eval loop.

  • The eval can’t see order. grade_row_set is “F1 over key values, order-insensitive” (eval_graders.py:58); grade_series is per-point tolerance. Nothing scores x-axis order → no learning signal at all.
  • Goldens aren’t weights. They’re a benchmark the judge scores against — correcting one golden doesn’t change how the agent builds the next chart.
  • Doesn’t generalise. A fix on this golden says nothing about a rent-by-bedroom chart, or a new suburb question.
  • At generation time the agent still has no band-order fact to use — so it keeps sorting alphabetically.
Recommendation. Band order is a deterministic property of the data, not of one answer — so it belongs in the data-knowledge layer (a shared ordinals registry) and is applied by the chart skills. That’s exactly the layer you pointed at (“skills capture information about data and inform the agent”). The golden’s job is to prove the chart renders right — the eval loop is order-insensitive and structurally cannot teach ordering, so golden-only is a dead end for this class of problem.

4b · Curating & verifying it from the Golden tab — your two questions

Moving the fact into the data-knowledge layer only works if the curator can see it, edit it, and prove it from the Golden tab — not by editing Python. Two additions make that real, and the verify loop is lighter than it looks.

A Data-knowledge panel in the sandbox, mirroring the skills catalogue (same 3-level disclosure). Scoped to the golden's dataset — so area_band under nsw_sales and any other dataset are edited independently (your Q2).

② prove it with a Run — NOT a pipeline rebuild
  • Band ORDER is a presentation sort the chart skill applies over the already-extracted rows. To confirm an edit, press Run on the object in ② Sandbox — the fast per-object cascade — and the chart re-renders in the new order.
  • You do NOT re-run the dbt data lineage for an ordering change. The skill reads the ordinals at run-time (DB → seed), so edit → Run → verify stays entirely inside the Golden tab.
order ≠ values. Re-ordering bands = a sandbox Run (seconds). Changing what the bands are (the bucketing) = an upstream dbt/pipeline change — the ~3M-row data lineage. The UI should keep those two edits visibly distinct.

Net: the data-knowledge layer is authoritative, but it becomes a first-class, curator-editable surface in the Golden tab — you never touch agent/ordinals.py to fix a band order, and one Run proves it.

5 · Risks & open questions

  • Auto-sort surprises a curator who wanted a different order → the explicit “Sort x-axis” control + x_order param always win.
  • Unknown bands / new datasets → registry defines placement (unknown last); an unrecognised column falls back to today’s behaviour (no change).
  • SQL vs chart layer. Should the extract’s ORDER BY also change (a CASE expression), or is chart-layer sort enough? Recommend chart-layer — presentation order shouldn’t be dictated by extract SQL — but a CASE-order is a viable alternative for pure-SQL surfaces.
  • Numeric-parse fallback. For unregistered bands, “sort by leading number” is a decent generic default the control can offer, but the registry is authoritative for the two known columns.
