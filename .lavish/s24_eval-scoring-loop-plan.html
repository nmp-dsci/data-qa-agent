Turning golden answers into a versioned, scored, regression-gated flywheel — with a teacher agent that proposes improvements and has to beat the gate to land. This plan extends s14 (whose decisions D1–D7 stay locked) rather than replacing it; it adds the version-control spine, dev+prod capture, an N=1 scoring path, and the Evaluations tab.

02 · the loop

What we are building

One cycle: goldens are captured wherever the admin is (dev or prod), versioned into the repo, replayed against a fingerprinted agent build, scored, and gated. The teacher agent joins the same loop as just another proposer that has to pass the gate.

The three things that make this a loop rather than a test suite: the pack is versioned so a score is reproducible; the agent build is fingerprinted so an improvement is attributable to one lever; and the gate is symmetric — it blocks a regression as readily as it admits a gain.

03 · version control

The LLMOps spine: what is versioned, and where truth lives

Your requirement is “full version control LLMOps”. Concretely that means four independently-versioned artifacts, all stamped onto every score, so any number in the dashboard can be traced back to the exact inputs that produced it.

Versioned artifact
pack_versioncontent hash of the golden pack YAML — what we measured against
agent_versionprovider + model_id + prompt_hash + skills_hash + knowledge_version + image_tag + git_sha — what we measured
judge_prompt_hash
judge_model		frozen rubric + judge identity — how we measured
eval_run.idjoins all three to per-case results and back to query_runs traces
Source-of-truth rule

The repo is the source of truth. The database is a working surface.

  • Admins author in whichever DB they are in (dev or prod) — that is the ergonomic bit.
  • make eval-export serialises app.eval_cases → evals/cases/<dataset>.yaml.
  • That YAML lands in a PR and gets reviewed — a golden is a specification, so it deserves the same review as code.
  • make eval-import seeds any environment from the pack: a fresh clone, CI's ephemeral Postgres, or prod.

Without this, a golden authored in prod is invisible to CI — which is exactly why evals cannot gate anything today.

Pack shape — evals/cases/nsw_yield.yaml
# pack_version is the content hash of this file, recorded on every eval_run
pack: nsw_property
dataset: nsw_yield
cases:

06 · scoring

It has to work at N = 1

Your explicit requirement, and the right one — a loop you cannot run until you have 80 cases is a loop you will never start. The runner takes a filter and scores whatever it matches, down to a single case.

# the whole pack
make eval

# one dataset · one tier · one case — the inner diagnose→fix loop
make eval DATASET=nsw_yield
make eval DATASET=nsw_yield TIER=T2
make eval CASE=yield-postcode-trend-001

Two things degrade gracefully rather than erroring when the corpus is tiny, and the report says so out loud:

  • The holdout slice is skipped below ~10 cases and the run is labelled generalisation: unproven. A holdout of one case is theatre.
  • The regression gate still applies — “no case flips pass→fail” is meaningful at any N, including one. This is the half of the gate that works from day one.

Why this matters beyond convenience

Scoring a single golden end-to-end is what proves the wiring: pack → runner → real /ask → graders → judge → eval_results → gate → dashboard. Once one case flows through, adding the 80th is data entry, not engineering. M2 ships when one yield golden scores end-to-end.

08 · gates

Where it runs — and what blocks a merge

s14's decision D4 stays locked: the expensive scored run is not a PR gate. Two tiers, different costs, different jobs.

SurfaceWhat runsCostBlocks?
Every PR
ci.yml		 Deterministic only — zero LLM calls. tests/test_eval_pack.py: pack schema lint, unique ids, EXPLAIN every golden_sql against the migrated schema, validate the report contract. Catches golden rot for free. free yes — blocks merge
Local / manual
make eval		 The real loop. Full pack (or one case) through real /ask with channel=eval → G1–G4 incl. judge → persisted, stamped, gated. single-digit ¢/run developer judgment
Nightly
eval-nightly.yml		 Same scored run on a schedule, alerting on pass-rate drop. Reuses the local entrypoint and the GH-secrets pattern the deploy smoke test already uses. ~¢/night alert, not gate decision D-4

The pack-lint gate is the sleeper value here: a golden that cannot regenerate its own answer fails CI before it is ever used to grade the agent. It costs nothing and it is what stops the corpus rotting as the schema evolves.

09 · auto-research

The teacher agent

The part you called auto-research. It is deliberately last, because it is the only component whose correctness cannot be checked by inspection — it has to be checked by the gate, which must therefore exist and be trustworthy first.

step 1 — read-only diagnosis

Safe, and useful immediately

A Claude Agent SDK agent with read access only to eval_results + joined query_runs traces. It clusters failures by tier × grader × tag and reports hypotheses. A human picks the lever. Useful from the very first scored run; cannot break anything.

step 2 — propose one lever

Gated, auditable

Same agent, now permitted to write exactly one intervention — a knowledge page, a skill, or the prompt — then trigger a candidate run. The composed hash proves it moved one lever. The gate decides. An accepted change arrives as a PR carrying docs/evals/cycle-00N.md; a human reviews the diff.

Prerequisites this plan deliberately creates — the teacher is not safe without them: a broad-enough pack, a calibrated cross-family judge, the agent_version stamp, per-run cost caps, and one-change-per-cycle discipline.

Rebutting “eval-tuning theatre”

The obvious objection to an agent that optimises against its own scoreboard is that it learns the scoreboard. Three structural defences, all of which must be in place before step 2: the intervention is one auditable file tied to a hash; ~20% of cases are a holdout that never steers a fix but is still scored; and the regression gate makes “fixed yield, broke trends” impossible to ship silently.

12 · risks

Failure modes and migration concerns

Prod goldens can leak PII into git

A golden captured from prod carries a real user's question and real result rows into a version-controlled file. For a product whose whole thesis is RLS governance, that is the sharpest own-goal available. Mitigation: export redacts by default — questions reviewed, golden_data capped and scrubbed, as_user mapped to a seeded test identity, never a real user id. Gate it in the pack-lint test.

Judge self-preference

A DeepSeek agent judged by a DeepSeek judge scores itself generously. Mitigation: cross-family judge (Claude judging DeepSeek), calibrated against ~10 hand-scored cases before it is trusted, rubric frozen and hashed.

Overfitting at small N

With 3–10 goldens, any intervention can be tuned to pass. Mitigation: the run is explicitly labelled generalisation: unproven below the holdout threshold, and no cycle report may be written from a run in that state.

Golden rot as the schema evolves

Marts change; golden_sql silently stops running and the agent gets graded against a broken benchmark. Mitigation: the free PR gate EXPLAINs every golden on every PR. This is why M3's cheap gate lands before the corpus grows.

Backwards compatibility

1,137 existing query_runs have no agent_version_id. They stay null — the column is already nullable, so no backfill and no migration risk. Trend charts start at the first stamped run and say so rather than implying history.

Cost drift from the loop itself

A teacher agent that can trigger runs can burn budget. Mitigation: per-run cost cap, one candidate run per proposal, and channel=eval excluded from the user-facing daily caps but counted separately so it is visible.
