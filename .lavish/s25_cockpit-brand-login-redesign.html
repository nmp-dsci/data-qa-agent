Flight Deck — the Data Pilot cockpit brand

Data Pilot is a data-science front end: plain-English questions in, boardroom-ready insight reports out — governed by RLS, tuned to your business by your admins, with Explore and a SQL editor for hands-on work. 02 · The brand system

Night Flight, taken seriously

The token layer (s17) already names the identity — navy-ink cockpit neutrals, instrument gold. The brand work is not a new palette; it is committing to the cockpit: one kinetic signature, one instrument voice, one scene grammar, reused everywhere. No new fonts, no image assets, no repaint.

Navy ink#0a0d15 · --bgThe night sky every scene hangs in. Unchanged.
Instrument gold#d9a84e / #f2ca79 · --accentThe brand metal: plane, routes, lit waypoints, primary actions. Unchanged.
Sky blue#7fb0ff · --accent-2Second aircraft: links, focus, comparison cohort, secondary routes. Unchanged.
HUD phosphor#9ece6a · --hud (locked ✓)A new meaning, not a new pigment — the same green as --good. Live telemetry only: annunciator lamps, streaming readouts, scene symbology. Never static UI.
Dawn horizonscene recipeLayered glow + 1px horizon + perspective ground grid. Login at full strength; whisper version behind app gutters.

Type — the instrument voice

No new faces. Inter keeps prose and titles; JetBrains Mono is promoted from “code font” to the instrument voice — a formalized style for everything the cockpit “says”:

Waypoint · Eyebrow · Readout — 10px / 700 / +0.14em
$1,182,406 +33.1%

Rule: mono-caps-letterspaced text is reserved for labels, telemetry and section markers. Numbers comparing anything are tabular. Prose never wears the costume.

Motion — things fly, instruments sweep

  • Fly: the plane glyph always moves along a path (SMIL mpath for ambient loops, offset-distance transitions for stepped hops). Never a straight-line translate. Supporting traffic flies other airways at other speeds and directions (26–47s); the hero stays biggest and brightest.
  • Sweep: radar conic loops (exists in chat hero) and dash-march on routes (exists) stay the ambient loops, 6–26s.
  • Read: HUD numbers jitter inside realistic bounds (GS ±12 kt, FL ±4) on tabular numerals so nothing shifts layout — the cockpit feels airborne, the UI never wobbles.
  • Land: UI state changes keep the existing 120/180/260ms token family.
  • All decorative motion freezes under prefers-reduced-motion — the parked-plane composition is designed, not broken.

Voice — one metaphor per surface

  • Aviation words appear in instrument-voice labels only (WAYPOINT, PREFLIGHT, IN FLIGHT); buttons and prose stay plain English — “Run query”, never “Take off”.
  • Outcome first, mechanism second: “Share it the moment it lands”, then “governed SQL”.
  • Errors and empty states direct, never decorate: an empty Explore says what to run, in the interface’s voice.

06 · Every screen type, first-class

Responsive, accessible, weightless

Breakpoints

  • ≥1200 — full scene: 3 routes, ground grid, glass story panel (620px column).
  • 861–1199 — 2 routes, HUD strip hides, same split.
  • ≤860 (stack) — card first (sign-in above the fold), vertical route threads card → story; blur drops for perf; auto-advance untouched.
  • Short laptops (≤740px tall) — scene anchors to edges; card never collides with the HUD strip.
  • Touch — dots ≥44px hit area (padding, not size); auto-advance pauses on touch; 16px inputs (existing rule).

Accessibility

  • Scene is one aria-hidden layer; the story remains a keyboard-navigable role=tablist (arrow keys, Home/End — existing contract).
  • prefers-reduced-motion: plane parks at the last waypoint with full contrail — a designed still, not a stripped page. Visual-baseline screenshots stay deterministic.
  • Contrast: waypoint labels move from --faint to --muted on the glass panel; annunciator text ≥4.5:1; axe-clean login test kept green.
  • Light theme is a real variant, not an afterthought: the same scene reads as an aeronautical chart on chart-paper cream (routes in deep gold, grid in warm gray). Tuned in its own pass.

Weight & contracts

  • Still ~0 KB of assets: pure CSS/SVG, no images, no new fonts, no libraries.
  • SMIL animateMotion + CSS offset-path both ship in all evergreen browsers; fallback is the current translate hop.
  • e2e kept: login(page, "Admin") profile-button names, dot tablist roles, login_walkthrough_view / login_success events, axe-clean check. Visual baselines regenerate once — expected churn.
  • Backdrop blur limited to two panels, disabled ≤860px (mobile GPU cost).
07 · Build plan

One PR, five workstreams — the whole brand lands at once

Locked: every phase ships in a single pull request on s25-flight-deck-brand, so the app is never half-branded. The workstreams below are commit boundaries inside that PR, ordered so each one builds on the last and the branch is reviewable commit-by-commit. The Stream column in the matrix above says which one every screen lands in.

W1

Tokens & the Canopy the login

--hud token + instrument-label/hud-box/annunciator CSS, PlaneGlyph in icons.tsx, then the login itself: scene layer, full-screen route + Sortie, crossing traffic, live HUD readout jitter, reticle card, preflight annunciators, 4-waypoint story + capstone copy with the ASK mini-report, responsive + reduced-motion, ultrawide cap, corner-cut fix. Files: styles.css, icons.tsx, Login.tsx.

W2

The kit

FlightPath component (route + waypoints + optional flying glyph, progress prop) and the shared primitives extracted so every later stream consumes them rather than re-rolling markup. Files: ui/, styles.css.

W3

Chat — hero & in flight

Hero suggestions become a numbered flight plan; the SSE streaming state becomes the flight strip (PLAN → SQL → ANALYZE → COMPOSE → LANDED) over the same progress data, ghost pages intact. Files: ChatPage.tsx, ResultView.tsx, styles.css.

W4

Screen sweep

The eight remaining matrix rows: report-page section markers + KPI corner ticks, Explore contrails and empty state, SQL annunciator bar, Goldens chips, Evals hud-box scores + lamps, Admin health lamps, Settings NIGHT / DAY / AUTO relabel, shell ground-grid band. Mechanical, one commit per surface.

W5

Day theme, a11y & baselines

Light-theme scene tuning (the aeronautical-chart variant on chart-paper cream), contrast + axe pass, reduced-motion stills, perf check on low-end mobile, then regenerate Playwright visual baselines once for the whole branch and refresh the .lavish screenshots.

✓

Ship

Run /no-mistakes over the branch — review, tests, lint, docs, push, PR, CI — then merge to main, which triggers deploy-aws.yml. Verify the GIS button styling on the real Google Sign-in in prod once deployed.

Risks & honest costs

  • PR size — the cost of shipping it all at once: this is the one real downside of the locked scope. A brand-wide diff is harder to review and harder to bisect than four small ones. Mitigations: strict commit boundaries (W1–W5 above), no behavior changes riding along, visual baselines regenerated in exactly one commit so the diff noise is isolated, and each screen’s change kept to tokens/markup rather than logic.
  • Scene overwhelm: the biggest visual failure mode is the login reading “busy”. The cap is three aircraft — one hero Sortie plus two dim traffic planes on crossing airways — with hierarchy held by size and brightness, glyph smudges deleted, and the canopy vignette pulling eyes to card + story. If it ever reads busy, traffic goes first.
  • GIS button: Google’s iframe button cannot adopt the brand; the card design treats it as a framed foreign instrument (label above, annunciators below). Verify prod styling right after deploy.
  • Metaphor fatigue: guarded by the voice rule — instrument labels only, plain-English controls. If a tester ever asks “what does DIG mean?”, the waypoint word changes, not the structure.
