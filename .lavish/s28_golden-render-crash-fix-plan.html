P0 · render crash P0 · pack corrupts chart rows P1 · deterministic object builder frontend + eval_pack + agent
The one-line cause of the blank screen

A chart object's own render rows were replaced with a {_truncated,_head,…} digest by the pack exporter, that digest round-tripped back into the database, and Trend.tsx does for (const row of data.rows) over it with no guard and no error boundary anywhere in the app — so one bad object unmounts the whole React tree.

Your two questions, answered: (1) Correct — no LLM for defined charts. You submit the config deterministically, and the backend generates both the SQL extract and the sandbox transform deterministically too (via canonical_extract_sql + build_object_code, not the LLM) — it just re-runs them against live data to serve the object. Same config → same object, every time. The LLM stays only as a fallback for a request no defined type can express. (2) Yes — a base metric like n_sold ("properties sold") is augmented into "% of properties sold by X" through the measure's how ▾ modifier (count · sum · avg · % share-of-x · growth). The builder computes the within-x share deterministically — that augmentation is exactly what the new share/growth measure kinds add, so any additive metric can become a share or a growth without a new column.

2 · Root cause

How a chart's rows became a dict — the corruption pipeline

The golden was authored fine. The corruption enters on the export→import round trip that the whole pack workflow depends on:

④ open golden
💥 white screen
Trend.tsx:33 for…of a dict → TypeError → no boundary → SPA unmounts.

This is two independent defects that happen to share the symptom. Either one alone is survivable; together they take down the tab.

Bug A · the blast radius (frontend)

Every chart renderer trusts data.rows to be an array, and nothing catches a throw:

  • Trend.tsx:33 · Combo.tsx:39 · Bars.tsx:54 · Choropleth.tsx:111 · DataTable.tsx:63,85 — all do for…of / .map on data.rows with no guard.
  • grep -r ErrorBoundary across the whole app → zero hits. Any render throw = white screen.
  • Your target chart, Line+Bar = Combo.tsx:39, has the identical gap.
Bug B · the corruption source (pack)

_cap_rows was written to shrink the ground-truth row dump. But _redact only ever applies it to golden_report and golden_objects (eval_pack.py:168) — the rendered fields. golden_data is popped to a digest first and never reaches it.

So the {_truncated,…} dict envelope is produced only for data the UI must render as an array — it is wrong in 100% of the cases it fires.

3 · The fix — root cause first, in four parts

Make the tab uncrashable, stop the corruption, then make Line+Bar reliable

P0 Part 1 — Make the Golden tab uncrashable frontend

Two layers so no single object can ever white-screen the app again, and existing corrupted data renders instead of crashing.

1a. Normalize rows at the chart boundary — coerce non-arrays, unwrap the _head envelope. New shared helper, used by every chart + the report dispatcher (frontend/src/ui/charts/tokens.ts + call sites).

+ // Tolerate any rows shape a stored/legacy report can carry: a plain array,
+ // the pack's {_truncated,_head} digest, or a missing/garbage value.
+ export function asRows(v: unknown): Record<string, unknown>[] {
+   if (Array.isArray(v)) return v as Record<string, unknown>[];
+   if (v && typeof v === "object" && Array.isArray((v as any)._head))
+     return (v as any)._head;               // capped digest → render the head
+   return [];                               // undefined / dict / garbage → empty
+ }

// Trend.tsx parsePoints, Combo.tsx, Bars.tsx, Choropleth.tsx, DataTable.tsx:
- for (const row of data.rows) {
+ for (const row of asRows(data.rows)) {

1b. Wrap each rendered object in an error boundary so a bad object shows a fallback card, not a blank page (frontend/src/report-engine/PageLayout.tsx — the ObjectBody dispatch at line 27, plus a new ChartErrorBoundary).

// PageLayout renders each object via ObjectBody(o) — wrap it:
- {renderObjectBody(o)}
+ <ChartErrorBoundary label={o.type}>{renderObjectBody(o)}</ChartErrorBoundary>
// ChartErrorBoundary = a 20-line class component (getDerivedStateFromError)
// rendering "⚠ this chart couldn't render" — the FIRST boundary in the app.
Why 1a fixes the current corruption for free: unwrapping _head means the 2 already-corrupted goldens render immediately (with their first 50 points) with no data migration — the tab is usable the moment this ships.

P0 Part 2 — Stop the exporter corrupting chart rows scripts/eval_pack.py

Keep the size guard, but never turn a rendered array into a dict. Truncate to a shorter array instead of the {_truncated,…} envelope, so every consumer still gets a list.

eval_pack.py · _cap_rows (line 133-139)

  if isinstance(node, list):
      if len(node) > MAX_GOLDEN_ROWS:
-         return {
-             "_truncated": True, "_total": len(node),
-             "_sha": _digest(node),
-             "_head": [_cap_rows(i) for i in node[:MAX_GOLDEN_ROWS]],
-         }
+         # Keep it an ARRAY — golden_report/objects are RENDERED, and a dict
+         # here is unrenderable. Truncate; drift is tracked by golden_data_sha.
+         return [_cap_rows(i) for i in node[:MAX_GOLDEN_ROWS]]
      return [_cap_rows(i) for i in node]

Then make eval-export to rewrite the committed pack in the clean shape, and add a defensive unwrap on import (Bug C insurance). A one-line test_eval_pack.py assertion — "no _truncated anywhere in a pack case" — locks it so this can't regress.

P1 Part 4 — The deterministic object builder: every type, every option defined upfront your steer

Per your direction: every defined object type is built deterministically from a structured config — no LLM. Each type exposes its complete option set upfront (dropdowns for finite choices, open text only where a value is genuinely free), so there is nothing to hallucinate. Setting the config and pressing Run regenerates the SQL extract + the sandbox transform to serve the data.

The anti-hallucination guarantee — every dropdown is bound to the schema's typed vocabulary: the Explore manifest (manifest.py) already classifies each dataset's dimensions (categorical / ordinal / time / geo) and metrics (additive / derived, with currency/number/percent format), and already ships them to the UI at /explore/datasets with their allowed values. So a user can only pick cuts & metrics the data actually has — a valid object is the only thing the form can produce.
The engine already exists for the 5 chart types. The deterministic path is wired end-to-end today (main.py:1176-1228): an empty instruction + a non-empty spec ⇒ canonical_extract_sql rewrites the SQL from the config and build_object_code emits the sandbox code — no LLM, and it already applies the last-N-months window. Builders exist for compare · breakdown · trend · kpi · table. This part is defining the option set per type + filling four capability gaps, not building an engine from scratch.

P0 Part 5 — Exhaustive builder tests: every object type, every option your steer

Per your check: a test matrix that builds every object type with every option and asserts it renders — so a dropdown that can't actually produce a valid object is caught in CI, not by you. The whole point of a finite, schema-bound vocabulary is that it's enumerable, so the test can iterate it exhaustively.

a
Backend matrix (deterministic, fast). Parametrize over every combination the manifest allows — {dataset} × {object_type} × {each metric × each "how" (count/sum/avg/share/growth/latest)} × {each dimension} × {group on/off} × {window} — call build_object_code + the extract, and assert per case: the object is non-null, data.rows is a non-empty array (never a {_truncated} dict), and the renderer's required fields for that type are present (e.g. compare ⇒ dimension + measure + line_measure; kpi ⇒ value; table ⇒ non-empty columns). Because it's deterministic there's no flake and no LLM cost — it can gate every merge.
b
Option-vocabulary test. Assert every dropdown option the UI offers for a dataset comes from the manifest (no hardcoded/dangling option) and every manifest dimension/metric is buildable — the two lists must match, so the UI can never present a choice the builder can't honour.
c
Render smoke (frontend). A component test feeding each object type a builder-produced object (and a deliberately malformed rows) asserts it renders a chart or the error-boundary fallback — never throws. This is the regression lock for the Part 1 crash.

Lives in services/data-agent/tests/ (builder matrix) + frontend/…/*.test.tsx (render smoke), wired into the existing CI jobs.
