Structured Builder · s29 plan

Metric = aggregation + a derived augmentation

Restructure the builder's metric row so each measure reads name = aggregation of column(s) · window · derive, and add a derive step that turns a base metric into a growth rate, a % of total, and more — computed deterministically over the window, no LLM.

Goal · current state · desired behavior

Today

One dropdown (how) mixes the base aggregation (sum / mean) with augmentations (% share / growth / latest). The augmentations replace the base rather than layering on it, and growth compares the window's first month vs its last month — two single months, not two periods.

Desired

Pick the aggregation (sum / average / weighted-avg) and its column(s) first — that's the metric. Then a separate derive dropdown augments it: % of total by group, growth over the window (last N months vs the prior N), and others. Growth of a 12-month window = this year vs last year.

Rule of thumb: % of total and cumulative are meaningful only for a sum (you can add up additive parts; you can't add up averages). Everything else works on any base, because it operates on the base value the aggregation already produced.

Growth semantics — the one behavior change to confirm

Now — first vs last month

Over a 12-month window: (month₁₂ − month₁) / month₁. Two single months — noisy, and sensitive to which months bound the window.

■□□□□□□□□□□■  →  compares just the two ■
Proposed — period over period

Over a 12-month window: (Σ/avg of the last 12 mo) − (Σ/avg of the prior 12 mo), ÷ the prior. The window length is the comparison period, exactly as you described (months 1–12 vs 13–24).

[■■■■■■■■■■■■][▨▨▨▨▨▨▨▨▨▨▨▨]  →  recent vs prior

Needs 2×window months of history; when there isn't enough, the metric returns 0/NaN for that key (same honest-failure posture as today's growth requiring month in the grain).

Risks & notes

  • Backwards compatibility — existing goldens store how: "share"|"growth"|"latest". The loader maps them onto the new {agg, derive} shape so saved objects keep working; growth ones will re-render with the new period-over-period math (values change — that's the intended fix).
  • Scope — every augmentation is deterministic pandas over the extract, no LLM and no new SQL. But each one is code + tests, so the set you pick sizes the work.
  • Non-additive guard stays — % of total / cumulative are offered only when the base is a sum of an additive column; the builder won't let you sum a stored average.
