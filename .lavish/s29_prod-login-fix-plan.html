s29 · prod incident diagnosis + fix plan · 2026-07-23

Prod Google login — one click, not five

Signing in on prod bounces back to the login card 3–4 times before it finally works. The diagnosis: login is the only thing in the app that cannot tolerate the database being asleep — and the database is asleep precisely when a user arrives. Every layer here is behaving as designed; the designs just collide at the front door.

worktree-prod-login-fix deqfc8b0u8s64.cloudfront.net AUTH_MODE=google Aurora Serverless v2 · min 0 ACU login_failure ×4–5 per session
Root cause — CONFIRMED (CloudWatch, 2026-07-23): Aurora cold start, hang variant. The cluster metric shows 0 ACU (paused) until exactly 21:59 UTC — the minute of the observed bad login — then 2.0. During resume, asyncpg connects hang rather than error: the page-load POST /events took 31s (sent :00.6 → 201 at :31.8) and the first credentialed /me hung ~19s. That is why there are no 5xx anywhere — failures never look like failures. The user experience: button appears dead (no pending state), so the user re-clicks 3–4 times; all queued logins land at once when the DB wakes (/me 200 ×4 in 4s) and the app enters; then a reload six seconds later wipes the in-memory-only session and forces one final login. Cold start × zero feedback × no session persistence = "4–5 attempts". Every mechanism is now log-proven; nothing here is conjecture.

2 · Failure mechanisms

What is actually failing — all three mechanisms, log-proven

M0 · Aurora cold-start hang — the trigger

ACU metric: 0 → 2.0 at exactly 21:59 UTC. While resuming, DB-touching requests hang silently ~20–30s (asyncpg connect waits; nothing errors): POST /events sent :00.6 → 201 at :31.8; first credentialed /me preflighted :12.9 → 200 at :31.97. All queued requests complete in a burst the moment the DB wakes — which is what the "/me 200 ×4 in 4s" really was. No 5xx ever appears, so logs look healthy while users suffer.

M1 · zero feedback during the hang — the multiplier

While getMe() hangs there is no pending state anywhere — the button just sits there. A hung login is indistinguishable from a dead one, so the user re-clicks 3–4 times; each re-click mints another credential and queues another hung /me. This is where "4–5 attempts" comes from. (Related, still real: in automation the button also swallowed clicks outright with no FedCM sheet and no request — E9 — and there is no GIS error callback registered to catch it.)

M2 · no session persistence — the re-toll

The session is a JS variable; the Google ID token hard-expires at ~1h with no refresh. The timeline shows the user fully inside the app at :32–:40 (conversations + admin dashboard fan-out) — then a reload at :41 wipes the token and dumps them back to the login card for one more round (:48). Every reload, new tab, or return visit re-pays the entry toll — including, on a cold visit, the M0 hang.

That is the lost-user scenario this ticket exists for. And the reason dev never reproduces any of this: local Postgres is always up, so the hang window simply doesn't exist.

Four aggravators (each individually reasonable)

A1 · session lives in a JS variable

In Google mode the token exists only in memory (api.ts:331) — every reload or new tab is a from-scratch login. This multiplies how often users stand at the front door, which is exactly where the cold-DB window bites.

A2 · NullPool, no retry — by design

db.py deliberately holds zero idle connections so Aurora can auto-pause (the dominant idle cost). The trade was priced for data queries, not for the login path where the user has no context and no patience.

A3 · Google ID token = the session

The raw ID token is the bearer for every request, and it hard-expires at ~1h with no refresh. Mid-session everything starts 401ing; combined with A1, "the app keeps logging me out" is guaranteed.

A4 · failure UX is a dead end

The only feedback is a small red Could not load profile (500). No "warming up" state, no auto-retry — even though the app is already holding a valid credential and needs zero further user input to try again.

4 · Fix plan

Phased: reliability first, then persistence, then polish

P0

Make the wake honest — pending state, classified waits, early warm-up ship first

No infra changes, no auth-model changes. Attacks M0+M1 directly: the moment the GIS callback fires the button flips to "Signing you in…", and because the failure mode is a silent hang (not an error), the client bounds the wait itself and narrates it — one honest "waking the warehouse" wait instead of N dead-feeling clicks.

F1 · backend: bound + classify the wait

Give the engine a connect timeout (~5s, connect_args in db.py) so a resume-window request fails fast instead of hanging 30s, and map connect-phase failures (refused, timeout, CannotConnectNowError) to 503 {"detail":"db_warming"} + Retry-After: 5 in the existing handler (main.py:58). Real 500s stay 500s; hangs become classifiable signals.

F2 · frontend: narrated auto-retry

Button flips to "Signing you in…" the instant the GIS callback fires (kills the dead-button re-click loop). getMe() retries on db_warming with ~4s backoff up to ~75s behind a "Waking warehouse · ~20s" state (mock above). The credential is already in hand — no second popup, ever. lib/auth.ts + Login.tsx.

F3 · wake the DB on card mount

Add unauthenticated GET /health/db (a SELECT 1 that swallows connect failures) and fire it when the login card mounts. In the observed session the FedCM dance took ~12s while the wake needed ~31s — starting the resume at first paint shaves the overlap; a returning-user's warm visit hides it entirely.

services/backend-api/app/main.py — sketch
# DB still waking ≠ server broken: retryable 503 the client can key on
WAKING = (ConnectionRefusedError, asyncpg.CannotConnectNowError, ...)
if _is_waking(exc):
    return JSONResponse(503, {"detail": "db_warming"}, headers={"Retry-After": "5", **cors})
frontend/src/lib/auth.ts — sketch
callback: (resp) => { setToken(resp.credential); void finishLogin(); }

async function finishLogin() {          // credential in hand — never re-prompt
  for (const waitedS of schedule(75)) {
    try { return onUser(await getMe()); }
    catch (e) {
      if (!isDbWarming(e)) return onError(e);  // real failures still surface
      onWarming(waitedS);                       // "Waking warehouse · ~20s"
      await sleep(4000);
    }
  }
}
P1

Stop re-logging in — real sessions for Google mode primary fix (M1) decision below

The headline fix now that the logs show M1 is the confirmed mechanism: exchange the verified ID token once for an app-issued session (the HS256 machinery dev mode already uses, 8h TTL), so reloads resume and the 1-hour Google expiry stops logging people out mid-session. Side benefit: drops the per-request JWKS verify + app.users upsert (E4) to once per session. Three approaches — pick one in §5.

P2

Login UX + funnel telemetry fast follow

Button "Signing you in…" state the moment the GIS callback fires (today: silence while /me is in flight); distinct copy for warming vs genuine failure; track() events for login_warming_shown, login_retry_n, login_recovered so the admin dashboard shows the funnel; investigate E9 (FedCM cooldown making the button silently inert) with real-user telemetry.

P3

Verification & guardrails done = proven

Unit tests for the 503 mapping + retry schedule; a stubbed-verifier journey test for the google path (real GIS is untestable headlessly); extend cloud_smoke.sh to assert /health/db and the 503 semantics; after the next organic cold login, confirm in CloudWatch that the old ConnectionRefused 500s are gone (needs aws sso login --profile data-qa first — E8).

rejected: keep Aurora always-on

Setting min_acu = 0.5 makes the whole problem vanish but costs ≈ US$43/mo idle (0.5 ACU × ~$0.12/ACU-h × 730h) against ~$0 today — the exact trade the NullPool design exists to win. A pinger (EventBridge cron) is the same cost wearing a disguise. P0 keeps scale-to-zero and spends the wake time on honest UX instead of failed logins. Revisit only if real multi-user traffic arrives.

5 · Decision needed

P1: how should a Google session persist?

  • Reuses create_access_token + _dev_user decode paths nearly verbatim
  • No cross-site cookie problem (CloudFront vs App Runner domains — the exact risk the code comment rejects)
  • Kills per-request JWKS verify + users upsert
  • Cost: token readable by JS — XSS exposure. Acceptable for a portfolio app; note CSP as follow-up. A cookie (option B) can replace the storage layer later without redoing the exchange.
  • Most secure (httpOnly) and deletes the whole CORS class of bugs
  • Same exchange endpoint as A — A is a stepping stone, not a dead end
  • Cost: Terraform + CloudFront work, VITE_API_URL rework, and SSE streaming (/ask/stream) through CloudFront needs explicit no-buffering care. Bigger blast radius — better as its own ticket.
  • No backend change at all
  • Cost: still a hard ~1h cliff mid-session; FedCM prompt cooldowns make "silent" unreliable (E9); per-request verify + upsert stays. Weakest option — listed for completeness.
