# Infrastructure (Azure, `dev`)

Bicep templates that deploy the app to **Azure Container Apps**, env-parameterized so `staging`/`prod` are the
same template with a different `env`. Nothing here is deployed automatically — review it, then run the
GitHub Action (or the manual commands) when you're ready.

## What gets created (`dataqa-dev-*`)

| Resource | Module | Purpose |
|---|---|---|
| Log Analytics | `monitoring.bicep` | Container Apps logs/metrics |
| Container Registry | `registry.bicep` | Holds the service images |
| User-assigned Managed Identity | `identity.bicep` | ACR pull + Key Vault read (no passwords in apps) |
| Key Vault (+ secrets) | `keyvault.bicep`, `secrets.bicep` | DB connection strings, JWT secret, Anthropic key |
| PostgreSQL Flexible Server (+pgvector) | `postgres.bicep` | The database |
| Container Apps environment | `environment.bicep` | Runtime for the apps |
| 3 Container Apps + 1 Job | `containerapps.bicep` | frontend, backend-api, data-agent, db-migrate |

The **db-migrate** job (`services/db-migrate/`) applies schema + RLS + seed, rotates the app-role passwords to
the Key Vault values, and loads the housing data against Azure Postgres. It's idempotent and the deploy
workflow runs it automatically after the apps deploy.

## Prerequisites (one-time)

1. **You run** `az login` and pick the subscription. *(I never see your credentials.)*
2. **GitHub OIDC** — create an App Registration with a federated credential for this repo, so the workflow
   deploys without any stored key:
   ```bash
   az ad app create --display-name dataqa-deployer
   # add a federated credential for  repo:<org>/<repo>:ref:refs/heads/main  (subject)
   # create a service principal, then assign roles on the subscription/RG:
   #   - Contributor              (create resources)
   #   - User Access Administrator (Bicep assigns AcrPull / KV roles)
   #   - Key Vault Secrets Officer (write the secrets)
   ```
3. **GitHub → Settings → Secrets and variables → Actions:**
   - **Variables:** `AZURE_CLIENT_ID`, `AZURE_TENANT_ID`, `AZURE_SUBSCRIPTION_ID`, `RESOURCE_GROUP`
     (`dataqa-dev-rg`), `LOCATION` (`australiaeast`), `NAME_PREFIX` (`dataqa`).
   - **Secrets:** `PG_ADMIN_PW`, `APP_USER_PW`, `AGENT_RO_PW`, `JWT_SECRET`, and optionally `ANTHROPIC_API_KEY`.
     *(These are generated by you and stored in GitHub — never pasted into chat.)*

## Deploy

**Recommended — GitHub Action:** run the **Deploy (dev)** workflow (`.github/workflows/deploy.yml`). It:
1. logs in via OIDC, ensures the resource group;
2. **phase 1** deploys the platform (`deployApps=false`) and reads the ACR + API URL outputs;
3. builds & pushes the 3 images (frontend baked with `VITE_API_URL`);
4. **phase 2** deploys the apps (`deployApps=true`).

**Manual (local):**
```bash
az group create -n dataqa-dev-rg -l australiaeast
export PG_ADMIN_PW=... APP_USER_PW=... AGENT_RO_PW=... JWT_SECRET=...   # your values
az deployment group create -g dataqa-dev-rg -f infra/main.bicep -p infra/main.bicepparam
# then build/push images to the created ACR and re-run with deployApps=true
```

## Local ↔ cloud config mapping

Same `APP_ENV=dev`; only *where config comes from* differs.

| Setting | local (`.env` / compose) | cloud (Container Apps) |
|---|---|---|
| `APP_ENV` | `dev` | `dev` |
| `AUTH_MODE` | `dev` (stub) | `dev` → `entra` later |
| `DATABASE_URL` | inline to local `db` | Key Vault `backend-database-url` |
| `AGENT_DATABASE_URL` | inline to local `db` | Key Vault `agent-database-url` |
| `DB_SSL` | unset | `require` |
| `JWT_SECRET` | dev default | Key Vault `jwt-secret` |
| `AGENT_URL` | `http://data-agent:8100` | internal app FQDN |
| `EXTRA_CORS_ORIGINS` | unset | the frontend's public URL |
| `ANTHROPIC_API_KEY` | `.env` (optional) | Key Vault (optional) |
| `VITE_API_URL` | `http://localhost:8000` | build-arg = the API's public URL |

## Migrations

The `db-migrate` image (`services/db-migrate/`) runs the schema/RLS/seed + housing load, records
`0001_phase0_init` in `app.schema_migrations`, and rotates the app role passwords every time it runs. It's a
Phase-0 stand-in; **Alembic** proper is the Phase-2 follow-up for richer incremental migrations.

## Cost & teardown

- Burstable Postgres + scale-to-zero Container Apps ≈ **$25–60/month** for dev.
- Tear it all down: `az group delete -n dataqa-dev-rg --yes --no-wait`.
